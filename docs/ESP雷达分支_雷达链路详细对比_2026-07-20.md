# ESP 雷达分支与当前主项目雷达链路详细对比

日期：2026-07-20

## 1. 范围、结论与证据等级

对比基准是 `分支项目/ESP 雷达`，目标是当前工作区顶层的
`ESPC51`、`ESPC52`、`ESPS3`。

**结论：当前主项目已经具备与基准分支相同的 LD2450 雷达主链路，
不是一个等待首次移植的空白链路。** C51 与 C52 的
`radar_ld2450`、`radar_ble`、`radar_domain` 目录逐文件比对无差异；
S3 的 `radar_ld2450`、`radar_ingest` 及绝大多数 `radar_domain` 文件
也一致。当前主项目仅在基准之上增加了房间人数落盘和习惯快照测试，
不会把 HOME 重新改为缓存源或活动源。

这是一份源码与静态差异审计：它证明目录、接口和源码行为的当前一致性，
不等同于本次重新完成了三块板、LD2450、BLE、UART、Wi-Fi 或服务器的真机回归。
用户给出的“基准分支已调通且正常”是迁移基准；本次没有 flash、monitor 或启动服务。

## 2. 完整数据链路

```text
每块 C5 的本地 LD2450
  -> BLE 通知回调 / C5 radar_ble 运行任务
  -> 固定 30 字节帧解析、边缘滤波、限频与可靠上传
  -> POST /local/v1/radar/result 到 S3

S3 自己的本地 LD2450
  -> UART 256000 8N1 / radar_rx
  -> radar_local_adapter

C5 远端状态 + S3 本地状态
  -> radar_gateway_ingest（身份、序列、格式、去重）
  -> RadarSourceState -> RadarRoomState -> RadarHomeState
  -> 坐标变换、分区、跟踪、短暂丢失连续性、诊断与本地 HTTP 状态接口
```

关键所有权不可改变：每块板均拥有一个本地雷达；C5 只产生自身的源状态，
S3 是空间解释、房间汇聚和 HOME 重算的唯一所有者。不能设计 UART 转发、
把 `active_source` 当 HOME，或以单个房间缓存替代 HOME 的重算。

## 3. C51/C52 边缘链路对比

| 层级 | 基准分支与当前主项目的共同实现 | 对等性结论 |
| --- | --- | --- |
| 帧解析 | `radar_ld2450/ld2450_parser.c` 固定帧头 `AA FF 03 00`，解析三个目标槽，损坏流逐字节重同步 | 相同 |
| 服务状态 | `radar_service` 将帧转成带 `local_id`、设备、房间与序列的样本，并记录 timeout/link state | 相同 |
| BLE 接入 | `radar_ble_runtime` 将通知投递给 domain；连接状态变化和超时也投递；BLE 启动失败记录后降级 | 相同 |
| 本地处理 | `radar_worker` 维护原始缓冲、边缘滤波、历史与 JSON 工作区，执行独立的上传任务 | 相同 |
| 发送边界 | C5 仅发 `/local/v1/radar/result` 的轻量结果，C5 不构造 Server 完整信封 | 相同 |
| C51/C52 身份 | 仅 `local_id`、`device_id`、`room_id`、绑定值不同，算法、帧格式和资源配额保持同构 | 相同，允许身份差异 |

源码证据：

- 基准 C51 的解析器在 `分支项目/ESP 雷达/ESPC51/components/radar_ld2450/ld2450_parser.c:7-12` 定义固定帧与重同步策略，并在 `:140-162` 解析三目标槽。
- BLE 回调、连接状态和 20 ms 状态轮询位于 `分支项目/ESP 雷达/ESPC51/components/Middlewares/radar_ble/radar_ble_runtime.c:23-54`；创建 2048 字节 PSRAM 栈任务在 `:58-83`。
- C5 domain 启动为雷达服务、原始缓冲、历史、JSON 工作区和两个任务分配资源，见 `分支项目/ESP 雷达/ESPC51/components/Middlewares/radar_domain/radar_worker.c:398-425`。
- 当前主项目与基准的 C51/C52 `radar_ld2450`、`radar_ble`、`radar_domain` 目录逐文件比较无输出，说明这些目录没有文件级差异。
- 共享协议头仍定义旧的 `/local/v1/radar/state` 常量，但当前 C5 worker 实际 POST
  `ESP111_PROTOCOL_ROUTE_RADAR_RESULT`，S3 仅注册 `/local/v1/radar/result` handler；迁移不可
  用旧常量替换运行路径。证据见 `ESPC51/components/Middlewares/radar_domain/radar_worker.c:293-300`、
  `ESPS3/components/esp111_protocol_common/include/esp111_protocol_common.h:152-154` 与
  `ESPS3/components/Middlewares/local_http_server/local_http_server.c:1107-1108`。

## 4. S3 链路对比

| 层级 | 已对等的责任 | 不能改变的迁移约束 |
| --- | --- | --- |
| 本地 UART 雷达 | `radar_ld2450` 管理 UART、配置 ACK、固定帧解析、presence 和 recovery | 保持 256000、8N1、缓冲尺寸、backoff 与诊断字段；不把 UART IO 放进 HTTP 回调 |
| 本地适配器 | `radar_local_adapter` 将 S3_LOCAL 源喂给统一空间域 | UART 初始化失败后记录 offline，仍保留 RX/recovery/adapter 任务；不能导致网关复位或误将 source 标为 online |
| C5 入站 | `radar_ingest` / `radar_gateway_ingest` 校验 JSON 上限、身份、local id、目标数组和序列 | 严格拒绝身份不匹配、倒退/冲突序列、畸形目标；重复应幂等 |
| 空间域 | source context、坐标、zone、tracker、person continuity、registry 和 spatial state | HOME 必须由当前在线 source contexts 重算；`source_person_count` 与 `home_person_count` 不可混用 |
| 对外可见性 | local HTTP radar handler、日志管理、诊断任务提供状态与调试观测 | 日志字段及接口必须与 Debug Tool/解析器兼容，且日志必须有界 |

源码证据：

- 基准网关按“网络与本地 HTTP -> 雷达 -> BME -> 轻量服务 -> 分析/调度 -> voice”分阶段启动；雷达失败只禁用模块而不阻断独立启动，见 `分支项目/ESP 雷达/ESPS3/components/Middlewares/gateway_orchestrator/gateway_orchestrator.c:71-140`。
- `radar_local_adapter` 先创建 `S3_LOCAL` source context、日志管理和 UART；UART 失败写入 offline 状态，随后为 adapter 创建 PSRAM 栈任务，见 `分支项目/ESP 雷达/ESPS3/components/Middlewares/radar_domain/radar_local_adapter.c:252-309`。
- S3 UART 服务使用独立 `radar_rx` 任务；UART 初始化失败返回可表示的状态且回收创建失败资源，见 `分支项目/ESP 雷达/ESPS3/components/radar_ld2450/radar_service.c:406-471`。
- 目标主项目和基准的 S3 `radar_ld2450`、`radar_ingest` 目录逐文件一致；`radar_domain` 仅有下一节所列的附加差异。

## 5. 当前差异及其影响

| 差异 | 当前主项目 | 影响判断 | 迁移处置 |
| --- | --- | --- | --- |
| `RadarRoomState.person_count` | 新增字段，并在 registry 从该源的 `source_person_count` 写入 | 补充房间快照；不改变 HOME 汇总的累加来源 | 保留，要求 host test 证明空闲源、在线源切换和多人合计不回归 |
| `radar_home_snapshot` | 当前主项目新增实现和 `test_habit_home_snapshot.c` | 为习惯规则提供 HOME 快照消费路径 | 保留为雷达域的下游只读消费者，不能反向缓存或修改 source/room 状态 |
| S3 构建与启动集成 | `Middlewares/CMakeLists.txt` 编译 home snapshot 并依赖 `habit_rule_engine`；orchestrator 在 radar 成功后启动 habit adapter/reporter | 使 HOME 消费者成为显式、受雷达结果门控的下游；不改变雷达输入、轨迹或 HOME 重算 | 保留此顺序；adapter/reporter 失败必须只记录并降级，不能阻断 BME、scheduler 或 voice |
| habit 事件上报 | 新增 `habit_event_reporter` 以 `/api/habit-events` 上报规则事件 | 是雷达派生的下游服务器行为，基准分支没有该任务和端点调用 | 不纳入“基准雷达核心对等”；保留时必须单独测量其 internal stack、HTTP 重试、离线堆积和创建失败 |
| host 测试脚本 | 当前主项目追加 habit snapshot 测试编译与执行 | 增强验证，非协议或算法替代 | 保留且纳入迁移门禁 |

准确差异：当前 `RadarRoomState` 比基准多出 `person_count` 字段
（`ESPS3/components/Middlewares/radar_domain/include/radar_registry.h:63`），
registry 在现有 HOME 累加前写入源人数
（`ESPS3/components/Middlewares/radar_domain/radar_registry.c:94`）。
测试脚本新编译 `radar_home_snapshot` 与习惯规则主机测试
（`ESPS3/components/Middlewares/radar_domain/tests/run_host_tests.sh:101-128`）。
此外，主项目在 `ESPS3/components/Middlewares/CMakeLists.txt:29,49,77` 注册该快照及其
依赖，并在 `ESPS3/components/Middlewares/gateway_orchestrator/gateway_orchestrator.c:94-99`
明确将 habit adapter/reporter 放在雷达启动成功之后。两项均为下游集成差异，非基准雷达
采集、解析或空间算法的替换。reporter 使用普通 `xTaskCreate` 创建 3072 字节任务，并在该
任务栈分配 JSON/响应缓冲（`ESPS3/components/Middlewares/habit_event_reporter.c:14-16,63-68,103-111`），
通过 `server_client` POST `/api/habit-events`（`server_client.c:1372-1385`）；这与基准雷达
PSRAM 任务模型不同，故必须作为额外资源风险，而非宣称为基准组成部分。

## 6. 资源调度与内存等价性

雷达功能对等还取决于调度，不可只复制协议文件：

1. C5 的 BLE 接收、雷达 worker、上传分别使用任务，雷达 worker 为 4096 字节、
   上传为 3072 字节、BLE 状态任务为 2048 字节；它们使用
   `MALLOC_CAP_SPIRAM | MALLOC_CAP_8BIT`。因此不能合并到音频或主启动任务，
   也不能无证明地移回 internal RAM。
2. C5 的雷达启动位于 BME 启动之后、C5 scheduler 之前，并且失败仅记录 warning；
   雷达并非系统启动的硬失败点。基准顺序见
   `分支项目/ESP 雷达/ESPC51/components/Middlewares/app_orchestrator/app_orchestrator.c:138-150`。
3. S3 先初始化 `resource_manager` 和 scheduler 的无任务队列，再启网络、本地 HTTP
   和雷达。雷达任务栈及 adapter/diagnostics 的大型工作区在 PSRAM，避免将其叠加到
   voice/HTTP 的 internal RAM 压力中；静态 mutex、计数器和其他同步控制对象则保持
   原有静态/internal 存储，不能笼统迁入 PSRAM。
4. 基准在雷达启动包络前后输出 `STARTUP_MEMORY_CHECK`，adapter、diagnostics 与日志任务
   有各自栈观测；`radar_rx` 当前没有 `app_stack_monitor` 输出。保留这些现有观测，若增加
   新任务则必须明确新增对应的栈/heap 验收，而不能误称为既有基线行为。

S3 雷达专属任务配额如下；同步时须将数值、优先级、内存能力和失败清理一并检查，不能
只比较任务名：

| 任务 | 栈 / 优先级 | 内存能力与失败语义 |
| --- | --- | --- |
| `radar_rx` | 4096 / 5 | PSRAM 栈；UART init 失败仍保留 RX/recovery，只有 task create 失败才 deinit UART 并返回 `ESP_ERR_NO_MEM` |
| `radar_local` | 8192 / 3 | PSRAM workspace 与栈；UART offline 仍保留 adapter，只有 task create 失败释放 workspace |
| `radar_diag` | 4096 / 2 | PSRAM snapshot 与栈；创建失败释放 snapshot |
| `radar_log` | 6144 / 1 | PSRAM 栈；失败返回错误，由上层决定雷达域降级 |

配额来源：`分支项目/ESP 雷达/ESPS3/components/radar_ld2450/include/radar_config.h:56-67,158-164`、
`radar_service.c:449-459`、`radar_local_adapter.c:285-303`、`radar_diagnostics.c:489-508` 和
`radar_log_manager.c:373-387`。

## 8. 对等性判定

当前工作区的雷达实现可判定为**源码结构、关键算法文件和任务资源策略与
`分支项目/ESP 雷达` 对等，且包含房间人数、HOME 快照及 habit 下游集成等 S3 扩展**。后续若要从基准同步新改动，
必须以本报告的全链路、身份边界和资源门禁为合同，而不是对整个目录进行盲目覆盖。
