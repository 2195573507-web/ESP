# 主项目雷达与完整 LCD 融合迁移及资源安全总计划

日期：2026-07-20  
文档状态：实施基线 / 多 Agent 协作总计划  
目标项目范围：主项目 ESPC51、ESPC52、ESPS3、ESP-server 及配套雷达调试工具  
输入基线：主项目当前资源分配、ESPLCD LCD 对比与实施计划、ESP 雷达分支详细对比与迁移计划

## 0. 执行摘要

本计划不是把五份文档机械拼接，也不是以“最小可用”为目标。最终目标是：

1. 主项目完整保留并持续对齐 `ESP 雷达` 分支的三源雷达能力：C51、C52、S3_LOCAL 各自拥有一个 LD2450，C5 完成 BLE 采集与边缘处理，S3 完成统一空间解释、跟踪、人员连续性、房间状态与 HOME 重算。
2. 主项目完整融合 `ESPLCD` 的 LCD 用户能力：ST7789 驱动、LVGL 页面、启动页、完整仪表盘、动画资源、命令文本、CST816T 触摸、语音状态和点击唤醒。旧 ESPLCD 中与 CSI 绑定的显示字段不删除，而是改由主项目雷达快照提供等价或更准确的占用、运动、人数和源状态。
3. 不恢复 Wi-Fi CSI。CSI 属于旧分支的数据来源，不属于 LCD 本身。其用户可见功能由雷达替代，CSI Kconfig、回调、任务、上传、算法和服务端字段禁止进入主项目。
4. 所有大栈、UI 对象、动画工作区、日志快照、雷达历史和 JSON 工作区优先放入 PSRAM；只有 DMA、cache-off 安全路径、极短控制对象和必须实时访问的控制栈留在 internal RAM。
5. 不以“编译通过”或“PSRAM 很大”作为资源安全结论。每个模块必须经过内存准入、完整初始化、发布、运行监控和失败回滚五个阶段；任何一步失败都不得留下悬空指针、重复任务、半注册回调、未释放 DMA 缓冲或错误的 ready 状态。
6. 多 Agent 允许并行复制和实现，但共享启动文件、公共协议、公共 CMake、资源管理器和最终合并只能由集成 Agent 串行处理。任何 Agent 不得递归覆盖整个工程。

本计划能提供的是可执行的资源安全设计和发布门禁。不存在仅凭静态文档就能绝对保证硬件永不内存不足的方法；因此“不会出现资源问题”的工程化实现方式，是把资源门槛、失败回滚、故障注入和真机并发证据设为发布硬门禁：达不到门槛就不发布该模块，而不是让系统在运行中崩溃。

## 1. 最终交付定义

### 1.1 雷达“完整”的定义

最终固件必须同时具备以下能力，缺一项不得标记雷达迁移完成：

- C51 与 C52 各自通过固定绑定的 BLE LD2450 接收通知，通知回调只复制字节和时间戳，不做解析、JSON 或 HTTP。
- C5 端固定 30 字节帧解析、流重同步、最多三目标、边缘过滤、历史、状态编码、独立 worker 与 upload task、有限队列和失败重试完整保留。
- C5 只向 S3 的 `/local/v1/radar/result` 上传自身源结果，不直连 ESP-server，不计算 HOME，不把目标槽位当作永久人员 ID。
- S3_LOCAL 通过 UART 256000 8N1 进入独立 `radar_rx` 和 `radar_local`，保留配置 ACK、presence、断线检测和恢复退避。
- S3 对 C51、C52、S3_LOCAL 执行身份、房间、序列、大小、目标 schema、重复、倒退和冲突校验。
- S3 的状态所有权保持 `RadarSourceState -> RadarRoomState -> RadarHomeState`，HOME 只从当前在线源上下文重算。
- 坐标变换、zone、tracker、短暂丢失连续性、人员 attach/recover、占用/静止/运动状态机、诊断和日志完整保留。
- `source_person_count`、`room.person_count` 和 `home_person_count` 分开维护，不互相覆盖。
- 调试工具能够区分三源、房间和 HOME，不把旧 fixture 字段误认为当前固件真实日志。
- 主项目新增的 habit HOME snapshot、环境告警和服务端上报只能作为雷达下游只读消费者，不得反向修改雷达域。

### 1.2 LCD“完整”的定义

最终 LCD 不是三行文字的最小版本，而是将 ESPLCD 用户可见能力融合到当前架构：

- ST7789 SPI2 驱动：初始化、复位、颜色顺序、反色、方向、X/Y gap、清屏、区域刷新、位图和颜色测试。
- LVGL 9.2.2 + esp_lvgl_port 2.6.2，完整启动页、主仪表盘、状态卡片、命令覆盖层、动画和页面切换。
- CST816T 触摸支持、坐标映射、点击/滑动事件；触摸硬件不存在时可观测降级，但目标硬件验收必须覆盖触摸。
- 猫咪/状态动画或 ESPLCD 等价视觉资源完整迁入；资源保存在 Flash/rodata，解码与工作缓冲放入 PSRAM，不允许全屏双 DMA framebuffer。
- Wi-Fi、BME690、雷达三源、房间/HOME 人数、运动/静止、语音状态、播放状态和网关状态都由只读 snapshot provider 提供。
- 旧 CSI motion 展示语义改为雷达 `presence/motion/person_count/source health`，界面功能不减少，但 CSI 实现不进入主项目。
- 保留 `display.show_text` 与 `lcd.show_text` 的现有命令入口，支持标题、正文、TTL、覆盖旧消息、到期恢复默认页。
- 点击唤醒通过主项目既有 `voice_chain_request_local_wake()` 或受控等价接口发出事件，不复制或覆盖旧分支语音链路。
- LCD 初始化失败、触摸失败、动画资源失败或 UI 内存不足时，Wi-Fi、BME、雷达、命令、语音和网关继续运行。

### 1.3 资源安全“完成”的定义

只有同时满足以下条件，才能宣称融合完成：

- C51、C52、S3 分别有新的 build map、启动内存日志、任务 high-water、并发运行记录。
- 任何一次初始化失败都无泄漏、无悬空回调、无重复任务、无空指针解引用、无 ready 状态误报。
- LCD 稳态只保留一个 4,800 B internal-DMA draw buffer；9,600 B legacy 行缓冲在 LVGL display 注册成功后释放。
- LVGL 对象、样式、动画元数据、页面模型、命令槽和图片解码工作区使用专用 PSRAM 分配器或显式 PSRAM arena。
- C5 voice 唤醒、录音、播放和 LCD 刷新并发时，internal/DMA 最大连续块不低于发布门槛，无 `ESP_ERR_NO_MEM`、I2S/LCD DMA allocation failure 或 cache 断言。
- S3 三源雷达、网络、环境告警、habit、voice proxy 同时运行时，无任务创建失败、队列无界增长、watchdog 或堆破坏。
- 所有资源门槛和失败路径都由代码执行，而不是只写在文档里。

允许先复制：

- 从 `ESP 雷达` 分支复制雷达基准目录到隔离 worktree，逐文件核对后形成补丁。

- 递归复制整个 ESPC51、ESPC52、ESPS3 或整个 `components` 覆盖主项目。
- 从 ESPLCD 恢复 CSI Kconfig、CSI service、CSI callback、CSI HTTP 或 CSI dashboard 数据源。
- 用分支的 app orchestrator、voice、BME、Wi-Fi 或系统命令整体覆盖主项目。
- 同时由多个 Agent 修改 `app_orchestrator`、`Middlewares/CMakeLists.txt`、公共协议头、资源管理器或 `sdkconfig.defaults`。
- 复制历史 build、managed_components 或 dependencies.lock 作为最终依赖结果。

## 3. 最终融合架构

```text
C51/C52 LD2450 BLE
  -> notify byte ingress
  -> radar worker / parser / filter / history
  -> bounded upload task
  -> POST /local/v1/radar/result
                                   \
S3_LOCAL LD2450 UART                -> S3 radar gateway ingest
  -> radar_rx / recovery           -> source contexts
  -> radar_local adapter           -> room states
                                   -> HOME recompute
                                   -> read-only snapshots
                                             |
                 +---------------------------+--------------------+
                 |                           |                    |
           habit rules                 diagnostics           ESP-server

C51/C52 read-only providers
  Wi-Fi + BME + radar source/home + voice + speaker + command
                 |
        bounded snapshot aggregator
                 |
 screen_service -> LCD adapter -> LVGL event mailbox
                 |
       ST7789 display + CST816T touch
```

关键设计：LCD 不直接访问 BME I2C、BLE、雷达 parser、HTTP 或语音锁；它只消费已经发布的固定大小快照。雷达不依赖 LCD，LCD 不参与 voice-exclusive ACK。LCD 刷新可以暂停或降频，但不能阻塞语音或雷达。

## 4. 资源治理总原则

### 4.1 内存能力分类

| 内存类别 | 允许内容 | 禁止内容 |
| --- | --- | --- |
| internal + DMA | LCD draw buffer、I2S DMA、ADC/DMA、必须 DMA 访问的 descriptor | UI 对象、图片、历史、JSON、大队列、整屏双缓冲 |
| internal control | 极短控制栈、静态 mutex/queue control、cache-off 必须路径、ISR 可见小对象 | 大任务栈、日志字符串、动画帧、网络响应缓冲 |
| PSRAM 8-bit | 雷达历史、JSON 工作区、网络大栈、UI 对象/样式/动画、图片解码、快照、非实时任务栈 | DMA buffer、ISR 直接访问、cache-off 必须访问对象 |
| Flash/rodata | 字体、图标、猫咪帧、静态文案、不可变查表 | 运行时可变对象、频繁写入状态 |

### 4.2 不允许用总空闲量代替最大连续块

每个关键分配点必须同时记录：

- `internal_free`、`internal_largest`
- `dma_free`、`dma_largest`
- `psram_free`、`psram_largest`
- 分配大小、能力标志、调用模块、初始化阶段和失败码

发布判断以 `largest` 为主、`free` 为辅。不能因为 `dma_free=35 KB` 就假设一定能分配 9.6 KB；也不能因为 PSRAM 尚有数 MB 就把 DMA buffer 放入 PSRAM。

### 4.3 碎片控制

1. 所有长期内存在启动后一次分配、长期持有，不在运行中反复 malloc/free。
2. LCD UI 使用一个固定 PSRAM arena；页面切换复用对象或从 arena 分配，不向全局 heap 高频申请小对象。
3. 命令文本使用固定容量 latest-only 槽，不为每条命令创建 LVGL 对象或动态字符串。
4. 动画只保留一帧解码工作区；源图片在 Flash，禁止预解码所有帧到 internal RAM。
5. 雷达 ingress、历史和上传 JSON 保持固定容量；失败重试不复制第二份完整 payload。
6. 日志使用有界 ring 或固定 scratch；禁止在高频路径格式化无界 JSON。
7. 初始化失败必须逆序释放；成功运行后不通过“反复重启模块”制造碎片，恢复优先重用已分配对象。

## 5. 量化资源预算与发布门槛

### 5.1 C5 LCD 新增预算

| 资源 | 启动瞬态 | 稳态 | 放置 | 规则 |
| --- | ---: | ---: | --- | --- |
| legacy LCD 行缓冲 | 9,600 B | 0 | internal DMA | LVGL display 注册后立即释放 |
| LVGL draw buffer | 4,800 B | 4,800 B | internal DMA | 单缓冲，禁止双缓冲/全屏缓冲 |
| LVGL task stack | 4,096 B | 4,096 B | internal control | priority 1；不在 timer 做 I/O |
| LCD bootstrap task | <=4,096 B | 0 | PSRAM | 初始化完成后自删 |
| UI arena | 64-128 KiB 初始上限 | 同左 | PSRAM | 对象、style、动画、页面模型；实际由统计收敛 |
| 图片解码工作区 | <=32 KiB | 按需复用 | PSRAM | 单工作区，失败则降级动画而非崩溃 |
| 命令/状态槽 | <=8 KiB | 同左 | PSRAM | 固定长度、双槽或 generation snapshot |
| 触摸状态 | <2 KiB | 同左 | internal/PSRAM 小对象 | 不创建高优先级独立大栈 |

完整 UI 可以增加 Flash 占用，但 factory app 仍以 5 MiB 为硬上限。字体、图片和动画超预算时，优先压缩资源、裁剪重复字体或转换图片格式，不允许通过增加 internal framebuffer 解决。

### 5.2 C5 内存准入门槛

初始门槛取保守值，并允许在两块板真实数据足够后调整；任何调整必须同时修改设计文档和自动化断言。

| 阶段 | 必须满足 |
| --- | --- |
| LCD bootstrap 开始前 | voice 处于 idle；`dma_largest >= 20 KiB`；`dma_free >= 30 KiB`；`internal_largest >= 20 KiB`；PSRAM arena 可一次性分配 |
| legacy 9,600 B 分配后 | 分配成功且记录地址/大小；其生命周期仅限 panel 到 LVGL 注册阶段 |
| LVGL steady 发布前 | legacy 已释放；`dma_largest >= 16 KiB`；`dma_free >= 24 KiB`；draw buffer 地址满足 DMA/internal 能力；UI arena 完整 |
| voice 唤醒前 | voice 资源管理器执行现有 DMA 准入；LCD 不再新增 DMA；若门槛不足只拒绝/延迟 LCD 刷新，不销毁 LVGL 对象 |
| 运行告警 | `dma_largest < 12 KiB` 或 internal largest 连续低于门槛时记录 CRITICAL，并冻结动画/降低刷新；不得继续新增 UI 对象 |

这些数值不是“理论上永远足够”的承诺，而是第一版硬门禁。根据现有 C5 日志中约 26 KiB 最大 DMA 连续块，它们比简单要求 32 KiB 更现实，同时为 4.8 KiB LCD 稳态和音频 DMA 保留余量。

### 5.3 S3 雷达与下游预算

S3 当前静态 D/IRAM 已使用约 70.1%，剩余约 102 KiB；因此不在 S3 引入 LCD 资源。雷达任务保持：

| 任务 | 栈/优先级 | 内存 |
| --- | --- | --- |
| radar_rx | 4096 / 5 | PSRAM stack |
| radar_local | 8192 / 3 | PSRAM stack + workspace |
| radar_diag | 4096 / 2 | PSRAM stack + snapshot |
| radar_log | 6144 / 1 | PSRAM stack |
| remote radar worker | 4096 / 3 | PSRAM，按构建路径启用 |

下游扩展必须单独治理：

- `habit_event_reporter` 的 JSON/response 缓冲不得继续长期占用 3,072 B internal task stack；改为固定 PSRAM workspace，任务栈根据 cache-off 审计决定是否迁移到 PSRAM。
- 环境告警 history、FIFO、completion queue 保持 PSRAM；队列满必须 drop/counter，不得阻塞 network worker。
- radar HOME snapshot 使用固定结构体，禁止复制整套 source histories。
- 新增任何任务前必须给出栈位置、最大栈高水位、队列深度、创建失败 unwind 和停止语义。

### 5.4 任务优先级与 CPU

| 级别 | 任务 | 原则 |
| --- | --- | --- |
| 最高实时 | speaker writer、mic/voice 实时路径、radar_rx | 不受 LCD 阻塞；不调用 LVGL |
| 高 | network worker、voice proxy、C5 dispatcher | 固定队列、有界处理 |
| 中 | radar worker/local、scheduler、command worker | 数据处理和状态机 |
| 低 | LCD LVGL、radar diagnostics、log、动画 | 可降频、可丢帧、不得反压实时任务 |

LCD UI 默认状态刷新 1 Hz，动画按 10-15 FPS 上限并允许在 voice active、Wi-Fi 重连、低 heap 或雷达高负载时自动降至 2 FPS 或静态帧。触摸输入保持 20-30 Hz 轻量轮询，只投递事件，不在回调执行语音或网络操作。

## 6. 指针、所有权与生命周期设计

### 6.1 统一模块状态机

所有新增或重构模块采用：

```text
UNINITIALIZED
  -> ALLOCATING
  -> DRIVER_READY
  -> SERVICE_READY
  -> PUBLISHED
  -> RUNNING
  -> STOPPING
  -> STOPPED
  -> FAILED
```

规则：

- 只有进入 `PUBLISHED` 后，其他模块才能取得接口。
- 任何失败从当前阶段按逆序释放，最后进入 `FAILED`；`FAILED` 不等于 ready。
- 所有 task handle、queue handle、timer handle、driver handle 和 buffer pointer 初始化为 NULL。
- cleanup 必须幂等；重复调用 stop/deinit 不 double free。
- 成功发布后，LCD 主 context 默认存活到重启，不在运行中释放后继续让 callback 持有旧指针。
- 需要重启模块时使用 generation token；callback 收到旧 generation 立即丢弃事件。

### 6.2 LCD 指针安全

- `lcd_runtime_t` 是唯一所有者，包含 panel、panel_io、spi bus ownership、LVGL display、indev、timers、queues、arena 和 buffers。
- `screen_service` 不保存 LVGL object pointer，只保存固定 adapter vtable 与 generation。
- 其他模块发布纯值 snapshot，不向 LCD 暴露内部字符串指针、队列项指针或雷达 track 指针。
- UI 更新由 LVGL task 在 lock 内执行。command task、touch callback、network callback 只能写固定 mailbox。
- 文本必须复制并保证 NUL 终止；所有长度在入口检查；UTF-8 截断必须按字符边界。
- 图片 descriptor 位于 Flash/rodata；解码输出归 UI arena；动画切换只替换索引，不 free 静态 descriptor。
- 触摸 callback 不直接调用 voice chain；它投递 `LCD_EVENT_WAKE_REQUEST`，由 app dispatcher 验证 voice state 后调用现有接口。

### 6.3 雷达指针安全

- BLE notify 和 UART RX 不保存调用者 buffer，必须立即复制到固定 ring/latest frame。
- JSON parser 结果必须复制为固定 target struct，不保存 cJSON/HTTP body 内部地址。
- source context 在 registry 生命周期内稳定；HOME snapshot 是值复制，不暴露内部锁保护指针。
- 目标槽位只代表本帧 slot；track/person ID 由 S3 tracker 管理，禁止跨源共享 raw pointer。
- queue 满、任务停止、设备断线时，旧 queue item 必须可安全丢弃，不执行已失效 context callback。

## 7. 启动顺序、内存峰值与回滚

### 7.1 C51/C52 最终启动顺序

1. `app_startup`：NVS、基础事件、Wi-Fi、gateway link、system command、memory/resource manager。
2. BME 与 C5 scheduler 基础结构。
3. 雷达 BLE/domain best-effort 启动；失败只禁用该源。
4. 初始化 voice control，但保持 idle，不在此阶段强制占用唤醒期 DMA。
5. `app_startup` 完成关键初始化并自删，释放其 12,288 B 栈。
6. `lcd_bootstrap` 在确认 voice idle 和内存门槛后启动：panel -> legacy buffer -> LVGL port -> 4,800 B draw buffer -> 释放 legacy -> UI arena -> UI/触摸 -> publish。
7. LCD ready 后启用动画与状态订阅；voice 可随时进入 exclusive。LCD 在 voice active 期间降低刷新，不释放/重建 DMA buffer。
8. command、radar、BME、voice、LCD 进入长期运行。

该顺序避免 `app_startup` 12 KiB internal 栈与 LCD 9.6 KiB 瞬态缓冲同时长期叠加，也避免在 voice 正在申请 DMA 时初始化 LCD。

### 7.2 S3 最终启动顺序

保持现有依赖顺序：Wi-Fi/event loop -> resource manager/scheduler queues -> network worker -> local HTTP -> radar registry/ingest/local -> radar downstream snapshot/habit -> BME/cache -> lightweight services -> diagnostics -> scheduler -> reporter/replay/device stream -> voice proxy。

任何可选模块失败都只禁用自己。禁止将雷达、habit、环境告警或 voice proxy 的失败改为全局 `ESP_ERROR_CHECK`。

### 7.3 失败回滚模板

每个初始化函数必须使用单出口或分阶段 cleanup label：

```text
allocate context
allocate arena/buffer
create queue
create driver
create task/timer
register callback
publish ready

failure:
  unpublish
  unregister callback
  stop/delete task and timer
  delete queue
  destroy driver
  free buffers/arena
  zero handles and state
  return explicit error
```

禁止先发布全局指针再继续初始化；禁止任务创建后初始化数据结构；禁止 task create 失败仅打印日志但保留 queue/buffer；禁止 deinit 后 callback 仍可进入。

## 8. 雷达完整融合实施

### 8.1 C51/C52

- 以 `ESP 雷达` 分支的 `radar_ld2450`、`radar_ble`、`radar_domain` 为基线逐文件同步。
- 保留各板自己的 device_id、local_id、room_id、BLE MAC/address type、硬件绑定和 sdkconfig。
- notify callback 只写 PSRAM ring/latest frame；worker 执行 parser、filter、history；upload task 构造固定 JSON 并发送。
- worker 4096 B、upload 3072 B、BLE task 2048 B 的既有配额先保持；只有 high-water 证据允许缩小。
- history 128 项、JSON workspace、任务栈保持 PSRAM。
- 失败路径要求：任一 buffer/queue/task 创建失败，释放之前已经创建的全部资源并返回可观察错误；BME、voice、LCD、gateway 继续运行。
- C51/C52 算法目录最终应零差异；身份配置差异必须显式列在 manifest。

### 8.2 S3_LOCAL 与远端 ingest

- 同步 UART service、parser、presence、config ACK、recovery，保持 256000 8N1 和当前引脚配置。
- UART init 失败进入 offline/recovery，不触发网关重启；task create 失败才完整回滚该模块。
- 远端 handler 严格检查 body 大小、device/local/room 身份、sequence、targets、坐标范围和重复。
- `/local/v1/radar/result` 是唯一运行 URI；旧 `/radar/state` 常量不得误用。
- source/room/HOME 统计每次从在线 context 重算；离线源按 TTL/continuity 释放，不污染其他房间。
- 保留主项目 `RadarRoomState.person_count` 和 `radar_home_snapshot`，但它们只读消费 registry。

### 8.3 参数与行为合同

- STILL_HOLD：2-3 秒范围，最终值通过统一配置定义，不散落宏。
- DORMANT：8-15 秒范围。
- reassociate：1.5-2.0 米。
- 新目标两帧确认、active-only 对外、速度异常拒绝、500 ms stale 与 1,500 ms 删除等当前行为不得因 LCD 融合改变。
- 不在同一次迁移中顺便修改 tracker 算法、房间映射或 BLE UUID；算法优化单独提交。

## 9. 完整 LCD 融合实施

### 9.1 驱动层

复制 ESPLCD C52 `lcd.c/lcd.h` 基线后重构为板级配置：

- SPI2，默认 SCLK 24、MOSI 23、CS 25、DC 26；RST、BL、分辨率、gap、旋转、反色、RGB/BGR 和 SPI 频率均来自 board profile/Kconfig。
- 目标面板 ST7789、240x284；首次硬件 bring-up 可用 10 MHz，验收后恢复 20 MHz。
- SPI bus ownership 明确：若已初始化则验证配置并共享，若由 LCD 创建则在失败回滚时释放。
- panel IO transaction queue 深度有界；flush completion 只通知 LVGL，不分配内存。
- DMA buffer 必须使用 `MALLOC_CAP_INTERNAL | MALLOC_CAP_DMA | MALLOC_CAP_8BIT` 并校验能力。

### 9.2 LVGL 与完整 UI

- 迁入 ESPLCD C51 的启动页、dashboard、猫咪动画和交互设计，但拆成 `lcd_driver`、`lcd_port`、`lcd_ui`、`lcd_assets`、`lcd_touch`、`lcd_adapter` 六个子模块。
- LVGL 依赖由 component manifest 锁定，不复制 managed_components。
- 使用单 240x10 RGB565 draw buffer；禁止 double buffer、full refresh 和软件旋转，除非后续资源报告证明安全。
- 将 LVGL malloc/free 接到专用 PSRAM allocator/arena；draw buffer 仍 internal DMA。
- UI 对象在启动时一次创建；页面切换隐藏/显示，不反复创建销毁。
- 动画资源静态存 Flash，解码使用单 PSRAM scratch；内存不足时冻结当前帧并记录计数，不影响主页面。
- 字体只打包实际字符集；中文字体使用子集化产物，避免完整字体撑爆 factory app。

### 9.3 数据提供者替换

禁止 UI timer 直接读取模块。新增固定结构：

```text
lcd_system_snapshot_t
  timestamp_ms / generation
  wifi_state / gateway_state
  bme_valid / temperature / humidity / iaq
  radar_sources[3] health / person_count / motion
  room/home occupancy / home_person_count
  voice_state / speaker_state / wake_allowed
  latest_command title/text/expiry
  alarm_level / degraded_flags
```

- 每个 provider 只复制当前已发布状态，最长执行时间有上限，不发 HTTP、不读 I2C、不等待 BLE。
- snapshot aggregator 以 1 Hz 合并，命令与触摸事件可即时触发 UI mailbox。
- 旧 CSI 卡片改名或复用为“空间状态/运动状态”，数据来自雷达。
- BME、voice、Wi-Fi 字段通过主项目现有状态 API 获取，不复制分支旧服务。

### 9.4 触摸和语音

- CST816T 使用 I2C0 时必须复用主项目 I2C 仲裁，不重新安装第二个 driver。
- 触摸地址、INT/RST 引脚、上拉、电平和旋转映射由 board profile 定义。
- 触摸读取有短超时；总线忙时跳过一帧，不阻塞 BME。
- 点击猫咪只投递 wake request；dispatcher 检查 voice idle、resource manager 和冷却时间后调用主项目 wake API。
- voice-exclusive 不要求 LCD ACK；LCD 只降低刷新和动画，显示最后状态。

### 9.5 现有命令入口

- `system_service_init()` 继续初始化 `screen_service`。
- `display.show_text` 和 `lcd.show_text` 仍进入同一 handler。
- `screen_service` 通过 adapter 绑定 LCD；未 ready 时返回明确 degraded 状态，不能空指针调用。
- 文本、标题、TTL 存入固定 latest-only slot；新消息覆盖旧消息；到期恢复 dashboard。
- ACK 表示“命令已接受并写入槽”，不声称物理 flush 已完成；物理显示失败通过独立状态上报。

退出：所有内存门槛、high-water、drop/retry、watchdog 和恢复标准通过。任何失败返回对应阶段修复，不通过扩大栈掩盖。

### 阶段 8：最终审计和发布

A8 对照合同进行只读审计；A0 生成最终资源表、功能矩阵、已知硬件配置和回滚点。

退出：无未知差异、无临时 bypass、无 CSI、无半初始化模块、无测试专用配置进入 release。

## 12. 必须执行的故障注入

对以下每个点分别强制失败一次：

- LCD context、legacy buffer、draw buffer、UI arena、decode scratch 分配。
- SPI bus、panel IO、panel driver、LVGL port、display、timer、touch 注册。
- LCD bootstrap/LVGL/touch event queue 创建。
- C5 radar raw buffer、history、JSON workspace、worker、upload、BLE task 创建。
- S3 radar RX/local/diag/log workspace 和 task 创建。
- network queue 满、HTTP 超时、Server 断开、BLE 断开、UART 无效帧。

每次失败必须证明：

1. 返回明确错误；
2. ready=false；
3. 已创建资源全部回收；
4. callback 不再进入；
5. 主业务其他模块继续运行；
6. 重试不会 double create/double free；
7. heap 和 largest block 回到允许误差范围。

## 14. 发布硬门禁

以下任一项出现，禁止发布：

- `ESP_ERR_NO_MEM`、task/queue/timer 创建失败后仍标记 ready。
- LCD legacy buffer 未释放、启用双 DMA buffer 或 UI 对象大量落入 internal heap。
- voice 唤醒后 DMA largest 低于门槛，或出现 I2S/LCD DMA 分配失败。
- 任意空指针、use-after-free、double free、heap corruption、stack overflow、watchdog、cache freeze 断言。
- C51/C52 source 混淆、HOME 使用 active source 缓存、source/home 人数覆盖。
- CSI 相关 Kconfig、符号、任务、回调或 HTTP 路径进入主项目。
- LCD timer 直接做 HTTP/I2C/BLE/雷达解析，或 command task 直接调用 LVGL。
- queue/retry 无界、离线期间持续复制 JSON。
- build 只在一个板成功，或使用旧 build 目录冒充新结果。

## 15. 回滚策略

- 雷达和 LCD 分别为独立组件提交；LCD 可整体关闭而不回滚雷达、voice、BME、Wi-Fi。
- LCD driver、UI、adapter、touch 分四层开关；某层失败只降级该层。
- 雷达同步按 C5、S3、本地 adapter、下游 snapshot 分提交，可回退到当前已知基线。
- shared orchestrator 改动必须有单独提交，便于恢复原启动顺序。
- 运行时不通过删除资源解决问题；资源不足时模块保持 FAILED/DEGRADED，主系统继续。
- 不使用 `git reset --hard`、`git clean` 或递归覆盖处理回滚。
