- 审计日期：2026-07-16
- 审计范围：`ESPC51`、`ESPC52`、`ESPS3`、`ESP-server`（仅接口契约）
- 审计模式：只读静态源码审计；未修改源码或配置，未执行 build、flash、启动 Server 或测试。
- 审计对象：当前工作树，包含尚未提交的雷达迁移与 CSI 删除改动；不等同于已发布固件。

## 0. 结论标记与总览

- **A 源码确认**：可由当前工作树直接证明。
- **B 静态推断**：由源码结构和固定参数推导，仍应以联调验证。
- **C 需要硬件验证**：静态源码无法证明，必须以真机、雷达原始帧和部署现场验证。

### 0.1 一句话结论

当前工程已具备 LD2450 的 UART 帧解析、C5 到 S3 的本地状态协议、S3 的三路雷达登记和诊断基础，但尚**没有** S3 到 ESP-server 的雷达上报链路，也没有能区分“静止有人”和“运动有人”的人体状态机。当前 `motion` 实际表示“连续看到目标”，不是依据速度的运动判断；因此不能确认雷达已经替代 CSI。**A 源码确认**

## 1. 完整雷达数据链路

### 1.1 当前实际数据流时序图

```text
LD2450（30-byte 报告帧，理论 10 Hz）
  |
  | UART 256000, 8N1, 无流控
  v
每个启用的 C5/S3: uart_driver RX ring 1024 B
  |
  | radar_rx 任务: 每次最多读取 128 B，阻塞最多 100 ms
  v
ld2450_parser: 0xAA FF 03 00 + 固定 30 B + 0x55 CC
  |
  v
radar_frame_t: 3 个 radar_target_t 槽位
  |
  v
radar_presence: UNKNOWN / MOTION / HOLD / VACANT_INFERRED
  |
  +-- C51/C52: radar_report 任务，JSON POST /local/v1/radar/state
  |             100 ms poll，最新 pending 快照合并
  |                 |
  |                 v
  |             ESPS3 radar_state_handler -> 严格 JSON 校验
  |                 |
  |                 v
  |             radar_registry[C51 或 C52]（按房间独立）
  |
  +-- S3_LOCAL: radar_local 任务每 100 ms 拉取本地 service 快照
                    |
                    v
                radar_registry[S3_LOCAL]

radar_registry / radar_diagnostics
  X  没有调用 protocol_adapter、sensor_aggregator 或 server_client
  X  没有 Server radar HTTP API / 持久化 / Dashboard 雷达字段接收方

现存 Server CSI 路径：ESPS3（当前未调用） -> POST /kernel/csi_event -> CSI cache/queue/DB
```

雷达链路在 S3 registry 止步，未到达“Server 上传”这一目标节点。**A 源码确认**

### 1.2 每一级数据格式和刷新频率

|级别|当前格式|刷新/超时|缓存方式|结论|
|---|---|---|---|---|

### 1.3 丢帧、阻塞、队列与恢复风险

|风险|评估|证据等级|说明|
|---|---|---|---|
|UART RX 溢出|中|B/C|1024 B 约为 34 个 30 B 帧；正常 10 Hz、128 B/100 ms 消费量充足，但无 UART event queue、无 overflow event 处理，且计数 `overflow_count/buffer_full_count` 从未实际递增。需压测。|
|解析器丢帧|低到中|A/C|半包和粘包按字节累积；坏帧丢首字节并重找帧头。100 ms 读超时会清空未完成帧，串口出现超过 100 ms 的异常帧间隙时会丢弃残片。|
|C5 到 S3 队列堆积|低|A|没有请求队列，只有一个 pending 最新快照；网络差时旧状态会被新状态覆盖，避免内存堆积，但丢失中间状态。|
|C5 HTTP 阻塞|中|A/B|单个 `radar_report` 任务可阻塞 HTTP 最长 3 s；不阻塞 UART RX 任务，但可能延迟下一次上报。|
|语音期间状态滞后|高|A|`radar_report` 在 `app_runtime_non_voice_is_paused()` 时不发送；语音独占暂停普通 HTTP/BME，因此雷达感知在上报层被暂停，恢复后只保留最新状态。|
|S3 本地锁阻塞|低|A/B|adapter、RX、诊断用短临界区或 mutex 拷贝小结构；无长时间计算。实际双核调度/峰值需验证。|
|错误后自动恢复|高|A|初次 UART 初始化失败只置 `unknown` 并返回；运行期 read error 后持续读取并延迟 100 ms，没有 UART deinit/reinit、指数退避、硬件复位或健康恢复状态机。|
|S3 到 Server 丢失|确定|A|不是重试问题，而是当前未实现雷达上行调用。|

|检查项|结果|证据等级|审计说明|
|---|---|---|---|

|串口参数|256000、8N1、无校验、无流控|A|符合代码配置；是否符合实际 LD2450 模块模式需 C 验证。|

|S3 UART 接线|配置为 UART1 TX18/RX17|A/C|默认配置存在；与实际硬件、其他外设复用和 TX/RX 交叉须真机确认。|
|帧头检测|有|A|字节滑窗匹配 `AA FF 03 00`，前缀不匹配逐字节丢弃。|
|长度校验|固定长度实现|A/B|以 `LD2450_FRAME_SIZE=30` 为唯一长度判定；帧头里的 `03 00` 没有单独解释/校验。若固件只输出该固定报告帧可接受，若混入其他数据帧会误判，需协议抓包确认。|

## 3. 人体运动判断算法审计

### 3.1 当前实际算法，而非预期算法

输入实际仅为：`target_count > 0`。速度、距离、角度、X/Y 位置、resolution、能量、信号质量均**不参与**状态判定。**A 源码确认**

```text
初始 / UART 不可用 / 3 s 未收到有效帧 -> UNKNOWN

最近 3 帧中至少 2 帧 target_count > 0 -> MOTION
MOTION 或 HOLD 中一旦又有目标             -> MOTION 并刷新 last_motion_ms
MOTION 持续 1 s 没有任何目标              -> HOLD
HOLD 期间连续无目标且距 last_motion_ms 15 min -> VACANT_INFERRED
```

注意：`last_motion_ms` 每次“有任意目标”都会刷新，不检查 `speed_cm_s`。所以 `MOTION` 的含义是“检测到目标持续存在”，`HOLD` 的含义是“目标消失后的短暂保持”，并不分别代表“有人运动”和“有人静止”。**A 源码确认**

### 3.2 问题逐项回答

|问题|回答|等级|
|---|---|---|
|当前是否真的可以判断人体存在？|只能判断“LD2450 输出了非零目标槽”；不能静态证明它一定是人体，也没有区域、质量或环境过滤。|A/B|
|静止人体是否可以检测？|若雷达持续给出一个目标，会保持 `MOTION`；若静止坐姿/睡眠导致目标消失，系统只会在此前状态上 HOLD，最长 15 min 后判空。没有呼吸微动检测。|A/C|
|进入延迟？|理论为 2 个有效帧，期望 100--300 ms；再加最多 100 ms C5/S3 poll。真实感知延迟受雷达报告率和网络影响。|B/C|
|离开延迟？|状态从 MOTION 到 HOLD 约 1 s；从 HOLD 到 `VACANT_INFERRED` 为 900000 ms（15 min），且要求出现空帧。|A|
|误报来源？|非人体目标、镜面/风扇/门帘、多径、邻室穿透、安装角度错误、非零坏数据均可成为“motion”。|A/B/C|
|漏报来源？|静止目标被雷达撤销、遮挡/盲区、超出配置距离、帧校验失败、100 ms partial reset、C5 UART 未启用、语音暂停上报。|A/B/C|
|阈值是否合理？|2/3 帧防抖和 1 s 短缺口合理作为起点；15 min 离开延迟不适合作为房间实时空闲判断，且没有现场标定依据。|A/B/C|
|是否需要状态机？|需要。现有枚举不是完整的人体行为状态机，缺少静止、进入、离开、质量衰减、失联和重新发现语义。|A/B|

## 4. 静止人体检测专项

### 4.1 部署场景判断

|人在房间内的状态|当前能否可靠保持 `occupied=true`|理由|结论|
|---|---|---|---|
|不移动但雷达仍持续输出 target|大概率会保持“有人”，但状态名为 `motion`|每帧有 target 就刷新 `last_motion_ms`。|A/B|
|坐着、微小动作|未知|取决于 LD2450 是否持续输出目标；当前算法没有微动质量判断。|C|
|睡觉|未知且风险高|没有呼吸/微动特征、床位区域策略或长期 track 保持。|A/C|
|呼吸|不能声明支持|目标结构和算法均没有呼吸频率、微位移时序或能量输入。|A|
|人已离开|会错误延迟为有人|空帧 1 s 后仍 HOLD，默认到 15 min 才 `vacant_inferred`。|A|

### 4.2 最低成本优化方案（不使用深度学习）

1. **先修正语义状态机**：将“检测到目标”和“目标速度超阈值”分开；`occupied` 由 track/置信度决定，`motion` 由速度或位置差决定。**B 静态推断**
2. **增加每目标短期关联**：以最近位置、速度和门限距离分配临时 `track_id`，保存 `last_seen`、`confidence`、连续出现次数。无需改变 UART parser。**B 静态推断**
3. **使用 occupancy timeout + confidence decay**：例如最后可信目标后 30--90 s 内仍 `occupied=true`，置信度随时间衰减；离开不再硬等 15 min。具体秒数需按房间实测标定。**B/C**
4. **增加静止证据窗口**：对于低速但稳定的 track，用位置稳定性、连续帧数和有效 resolution 作为 `STILL` 证据；不是将“任何目标”叫 motion。**B/C**
5. **呼吸微动仅作为可选 P2**：仅在供应商固件能够稳定输出足够精度的连续坐标/能量时，才在固定床位区域做低频微位移分析；当前数据模型没有质量字段，不能承诺可行。**A/C**
6. **卡尔曼滤波仅用于跟踪稳定化**：先建立带质量门控的关联和状态机；滤波不能弥补雷达根本不输出静止目标的问题。**B**

## 5. 人体目标跟踪审计

当前 target 为三个无 ID 的固定槽位；上报只包含位置、速度和 resolution。没有 `target_id`、时间戳序列、关联距离、滤波状态、生命周期或重新识别键。多人时槽位交换会被当作目标位置变化，无法判断同一人或新目标。**A 源码确认**

建议的最小生命周期：

```text
NEW --(连续 2--3 个关联帧)--> TRACKING
TRACKING --(低速度且稳定窗口满足)--> STABLE
TRACKING/STABLE --(短暂丢帧)--> LOST
LOST --(同一门控区域重现)--> TRACKING
LOST --(timeout 到期)--> EXPIRED
```

建议策略：

|主题|建议|等级|
|---|---|---|
|关联|同一房间内用预测位置与实测位置的门限匹配，拒绝跨房间合并。|B|
|离开|`LOST` 不立即清 occupancy；先做 3--10 s 目标丢失、再以 30--90 s occupancy 衰减决定空闲。|B/C|
|重新出现|仅在 `LOST` 窗口、空间门控和速度一致时复用临时 ID；超时后创建 NEW，不能声称跨长时身份保持。|B|
|多人|最多 3 目标是设备上限；须以一对一匹配和 unmatched 轨迹处理，否则槽位变换会混乱。|A/B|

## 6. C5 与 S3 职责建议

### 6.1 三个方案比较

|方案|优点|问题|评价|
|---|---|---|---|
|A：C5 解析 + 人体判断 + `occupied/motion` 上传|最少 S3 计算，断开 S3 时可本地判断|两块 C5 参数/算法易漂移；语音资源竞争；S3难以统一诊断和升级。|不推荐作为唯一判定层。|
|B：C5 原始摘要，S3 完整人体判断|统一算法与两房间策略|C5-S3 中断时丢失本地实时判断；摘要频率增加。|可行但需本地降级。|
|C：C5 过滤/最新摘要，S3 每源状态融合|C5 工作量小，S3 统一 track/状态机；每房间独立，并可保留 C5 的离线安全状态|需要新增 S3->Server 契约和显式数据所有权。|**推荐。**|

### 6.2 推荐最终架构

```text
C51 / C52（各自房间）
  UART parser -> 轻量范围/帧有效性门控 -> 最新 target snapshot + 本地短 TTL
  -> /local/v1/radar/state（latest-only、带源序列）

S3
  per-source registry -> per-room track association -> per-room presence FSM
  -> local dashboard snapshot + 可靠 latest-only Server uploader

ESP-server
  独立 radar.presence 契约/表或明确的 gateway snapshot `radar` 子对象
  -> 保留 `radar_state`、`occupied`、`motion`、`confidence`、`last_seen`、诊断计数
```

- C5 保留解析、帧健康、轻量异常过滤和离线短 TTL，不能在语音时停止 UART 接收。**B**
- S3 对 C51 与 C52 **分别**建立状态机，不将客厅和卧室目标做单一人体融合。**A/B**
- S3 是唯一 Server-facing 上传方，符合现有“C5 只连 S3 SoftAP”的边界。**A/B**
- C5 到 S3 断网时保留最新本地状态；恢复后上传最新快照和明确的状态变化/序列，避免补传所有帧。**B**

## 7. 资源与实时性审计

### 7.1 C5 新增资源

|项目|当前静态尺寸/优先级|影响评估|结论|
|---|---|---|---|
|UART ring|1024 B|小，但作为驱动内部缓冲通常占内部 RAM。|A/B|
|RX read buffer|128 B，位于 `radar_rx` 栈|小。|A|
|parser buffer|30 B|小。|A|
|目标/快照/presence|最多 3 targets + history 3 + mutex|小。|A|
|RX task|4096 B，priority 5|优先级高于 mic/voice priority 4；轻量 read/parse 一般合理，但需和 WiFi/音频真机测量。|A/B/C|
|report task|4096 B，priority 3|HTTP 最多 3 s，低于音频；语音 gate 会暂停它。|A/B|
|现有 mic|12288 B，priority 4|雷达加两任务后进一步压缩内部 heap/调度余量。|A/B|
|voice chain / response|各 8192 B，priority 4|与 radar RX 同时存在；当前策略优先保护语音上报而牺牲雷达上报。|A/B|
|speaker writer|6144 B，priority 6|不会被 radar task 抢占。|A/B|
|WiFi reconnect|3072 B，priority 5|与 RX 同优先级，需真机确认饥饿与 UART ring 高水位。|A/C|
|BME|由 runtime gate 暂停|雷达 RX 未被 gate；但 report 被 gate，BME 与 radar 上报均会滞后。|A|

当前没有对 radar RX task stack watermark、UART ring high-water、CPU 占用、连续 parse error、HTTP 失败时长做可导出的遥测；因此“不会影响 voice/mic/speaker/BME/WiFi”不能由源码证明。**C 需要硬件验证**

## 8. CSI 替代架构检查

|检查项|结果|证据等级|说明|
|---|---|---|---|
|C51/C52 CSI 业务代码|当前工作树中已删除|A|`csi_edge_detector`、`csi_phase_a`、`csi_placeholder` 处于删除状态。|
|S3 CSI 融合/占位代码|当前工作树中已删除|A|`csi_fusion`、`csi_placeholder_gateway` 处于删除状态。|

## 10. ESP-111 雷达人体感知链路优化建议

### 10.1 推荐状态机

```text
RADAR_OFFLINE
  --(连续有效帧)--> VACANT
VACANT
  --(新 track 置信度达门限)--> OCCUPIED_MOVING 或 OCCUPIED_STILL
OCCUPIED_MOVING
  --(速度/位置变化低于门限一段时间)--> OCCUPIED_STILL
OCCUPIED_STILL
  --(速度/位置变化恢复)--> OCCUPIED_MOVING
OCCUPIED_*
  --(所有 track LOST)--> OCCUPANCY_DECAY
OCCUPANCY_DECAY
  --(可信 track 重现)--> OCCUPIED_*
  --(timeout/置信度归零)--> VACANT
任意状态
  --(UART 故障或帧超时)--> RADAR_OFFLINE / UNKNOWN
```

`ENTER` 和 `LEAVE` 应是状态转换事件，而不是长驻状态；`occupied`、`motion`、`availability` 应作为彼此独立字段。**B 静态推断**

### 10.2 推荐数据协议（概念，非本次改动）

```json
{
  "schema_version": 2,
  "source": "C51",
  "room_id": "living_room",
  "sequence": 1234,
  "frame_seq": 5678,
  "timestamp_ms": 0,
  "availability": "online",
  "occupied": true,
  "motion": false,
  "confidence": 0.82,
  "last_seen_age_ms": 400,
  "tracks": [
    {"track_id": 7, "x_mm": 900, "y_mm": 2100, "speed_cm_s": 0,
     "distance_mm": 2285, "state": "stable", "confidence": 0.82}
  ],
  "diagnostics": {"parse_errors": 0, "uart_errors": 0, "rx_overflows": 0}
}
```

上述为推荐契约，当前源码尚未实现。`track_id` 是本房间短生命周期 ID，不应被解释为个人身份。**B 静态推断**

### 10.3 推荐阈值策略

|参数|建议初值|说明|等级|
|---|---|---|---|
|进入确认|2/3 帧|保留当前防抖思路，但输入改为可信 track。|B/C|
|运动速度|以实测噪声基线之上设置|不能在未验证单位/安装角度前固定数值。|C|
|静止确认|连续 3--10 s 低速且位置方差小|防止把短暂停顿误判静止。|B/C|
|track LOST|3--10 s 无关联帧|针对遮挡短缺口。|B/C|
|occupancy decay|30--90 s|比 15 min 更接近房间占用业务；睡眠场景需单独标定。|B/C|
|UART offline|保持当前 3 s 起点|结合重新初始化与实际帧周期验证。|A/B/C|
|区域门控|按房间安装高度、墙体/床位/门口配置|这是误报控制的前提。|B/C|

### 10.4 后续开发顺序

1. **P0 硬件基线**：确认每块 C5/S3 的 UART/GPIO/供电，抓取原始帧，验证帧格式、坐标、速度、实际刷新率和干扰源。**C**
2. **P0 闭环契约**：先定义 S3->Server `radar.presence`，补齐 S3 上传、Server 验证/持久化/Dashboard read model，再移除或退役 CSI 业务依赖。**B**
3. **P0 状态语义修复**：将 `occupied` 与 `motion` 拆开，缩短并参数化离开/衰减策略。**B**
4. **P1 可靠性**：UART 重新初始化、完整诊断、C5/S3 断链 latest-only 恢复、语音不暂停感知状态机。**B/C**
5. **P1 多目标**：S3 每房间 track manager、生命周期、区域门控和多人场景测试。**B/C**
6. **P2 静止/睡眠**：在真实床位/坐姿数据上评估静止置信度；只有硬件数据足够时再引入呼吸微动或卡尔曼滤波。**C**
7. **验收**：至少覆盖空房、走动、坐着、睡眠、两人、门口短暂经过、风扇/窗帘、S3 断链、C5 重启、语音长 turn、UART 拔插。**C**
