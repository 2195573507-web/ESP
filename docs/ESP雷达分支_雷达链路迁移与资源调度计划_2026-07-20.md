# ESP 雷达分支雷达链路迁移与资源调度计划

日期：2026-07-20

## 1. 目标与基线

目标是让当前主项目持续保持与 `分支项目/ESP 雷达` 完全相同的雷达功能：
三个独立雷达源（C51、C52、S3_LOCAL）、相同的 LD2450 帧解释、同一套 C5 边缘
处理和 S3 空间状态语义，以及不破坏 Wi-Fi、BME、语音、HTTP 与后台调度的资源模型。

当前审计显示主链路已迁入；本计划因此不是建议再次整目录复制，而是一个
**差异受控同步、构建验证和真机验收计划**。每一阶段的退出条件未满足前不得删除
旧实现、改变线程栈、改变身份绑定或声称已完成硬件迁移。

## 2. 不可协商的合同

| 合同 | 必须保持 | 禁止行为 |
| --- | --- | --- |
| 传感器拓扑 | C51、C52、S3 各自连接一个本地 LD2450 | UART 广播、跨板 UART 转发、把 C5 当 S3 UART 外设 |
| 帧与目标 | 固定 30 字节帧、最多三个槽、槽位不是跨帧人的永久 ID | 将槽号当人 ID；伪造或猜测人体身份 |
| 状态所有权 | `RadarSourceState -> RadarRoomState -> RadarHomeState`；S3 为唯一空间解释者 | C5 计算 HOME；缓存 active source/room 作为 HOME |
| 人数语义 | 源人数与 HOME 汇总人数分别记录 | 用 `source_person_count` 覆盖 `home_person_count` 或反向覆盖 |
| 本地传输 | C5 只发运行中的 `/local/v1/radar/result`；S3 验证身份、顺序与大小 | 误用仅保留常量的 `/local/v1/radar/state`、C5 直连 Server、放宽身份校验、将畸形 JSON 当成功 |
| 降级 | 雷达/BLE/UART 失败可观测且独立降级，网关与音频等独立模块继续运行 | 以 `ESP_ERROR_CHECK` 或无限重启绑定雷达可选启动 |
| 资源 | 大容量数据槽、工作区与雷达任务栈走 PSRAM；静态同步控制对象维持原有存储；HTTP/音频不得被雷达阻塞 | 把雷达接收/上传塞入 HTTP 回调或共享 voice 实时任务，或笼统将 mutex/控制对象搬入 PSRAM |

## 3. 实施阶段

### 阶段 0：冻结基线与建立差异清单

1. 以 `分支项目/ESP 雷达` 的 `ESPC51`、`ESPC52`、`ESPS3` 为唯一雷达基线。
2. 对四类路径分别作逐文件 diff：C5 `radar_ld2450`、`radar_ble`、`radar_domain`、
   app orchestrator/config/protocol header；S3 `radar_ld2450`、`radar_domain`、
   `radar_ingest`、local HTTP、gateway orchestrator、runtime/resource manager、main 和
   app config。
3. 同步 S3 `sdkconfig.defaults` 与 `radar_ld2450/Kconfig`，检查 UART enable、controller、
   TX/RX GPIO 与 raw-debug 开关；它们和源码共同定义实际硬件链路，不能只同步 C 文件。
4. 同步时检查 `CMakeLists.txt` 的源文件、include path 与 component dependency；没有被
   注册进 S3 `Middlewares` 的实现不视为完成迁移。
5. 每一处差异归为身份绑定、功能变化、资源变化、测试变化或下游只读扩展。未归类的
   文件不合入。
6. 当前已知允许差异：`RadarRoomState.person_count`、`radar_home_snapshot` 和其 habit
   下游集成；后者必须单列资源与服务器行为验收，不能混入“基准雷达核心完全相同”的结论。
   这些扩展必须只消费已重算的状态，不能修改雷达源状态机。

退出条件：形成无“未知差异”的清单；C51/C52 基准目录与目标目录继续为零差异。

回滚点：保留合入前的分支提交或补丁；绝不以删除 build 目录或重写未提交文件作为回滚。

### 阶段 1：C51/C52 采集和边缘域对等

1. 同步 `radar_ld2450` 的解析、service、codec、board config 及 host tests。
2. 同步 `radar_ble` 与 `radar_domain`，但仅替换算法/运行代码；保留各板自己的
   `local_id`、`device_id`、`room_id` 和实际绑定值。
3. 保证通知回调只投递字节；解析、滤波、JSON 和 HTTP 都在 worker/upload task 中执行。
4. 保证雷达启动失败只记录并降级：BME、scheduler、voice 与网关连接仍按原顺序启动。

静态门禁：

- `radar_worker` 继续有 4096 字节 worker、3072 字节 upload 的独立任务；
  BLE 状态任务为 2048 字节。
- 任务、历史缓冲、JSON 工作区均使用 `MALLOC_CAP_SPIRAM | MALLOC_CAP_8BIT`。
- 不新增从 C5 到 Server 的路径；本地 HTTP URI 与 payload schema 不变。

主机门禁：分别从 C51、C52 项目根执行其 `radar_ld2450/tests/run_host_tests.sh`。

### 阶段 2：S3 本地雷达与远端接入对等

1. 同步 S3 `radar_ld2450`：UART、config ACK、parser、presence 和 UART recovery 必须一起
   同步；不可只拿 parser。
2. 同步 `radar_local_adapter`、`radar_gateway_ingest`、`radar_ingest`、source context、
   coordinate transform、zone、tracker、continuity、registry、spatial state、diagnostics、
   log manager 与 radar local HTTP handler。
3. 保持输入验证：body 上限、`local_id`、设备/房间身份、序列、目标数组、重复和倒退
   状态的处理必须与基准一致。
4. 保留当前主项目的 `radar_home_snapshot` 作为 registry 之后的只读消费者；确认它不缓存
   source/room 来反向决定 HOME。
5. 若保留 habit rule adapter/reporter，必须在 `radar_local_ret == ESP_OK` 后启动；失败仅
   禁用该下游能力，不能改变雷达、BME、scheduler 或 voice 的启动和失败语义。

主机门禁：从 `ESPS3` 项目根执行
`components/Middlewares/radar_domain/tests/run_host_tests.sh`，并要求 gateway ingest、
ingest、spatial、person continuity 和 habit HOME snapshot 全部通过。该脚本不覆盖
`radar_local_adapter.c`；必须新增独立的 adapter start/failure/recovery 单元测试，或把它
明确列为 IDF build 加真机 UART 验收，不能称为该 host suite 已覆盖。

### 阶段 3：启动与资源调度门禁

1. S3 保持启动顺序：Wi-Fi/event loop -> resource manager/scheduler init -> network worker ->
   local HTTP -> radar registry/ingest/local adapter -> habit rule adapter -> habit event reporter
   -> BME -> command/lightweight services -> radar diagnostics -> scheduler -> replay/device stream
   -> voice。下游 habit 步骤只在雷达 local adapter 成功后启用。
2. 保留基准的 `radar.before` / `radar.after` `STARTUP_MEMORY_CHECK` 和已有 adapter、
   diagnostics、log 栈观测；若新增任务，才将对应 stack/heap 观测作为新增验收项。不得
   将目前没有该观测的 `radar_rx` 误报为已有 `app_stack_monitor` 覆盖。
3. 保持基准降级：UART init 失败后记录 `uart_offline`，仍创建 RX/recovery 和 local adapter
   以等待恢复；BLE start、queue/workspace 或 task create 失败才按各自源码已有路径释放
   已创建资源、禁用对应模块并留下可解析日志。不得把“完整 unwind”伪装为与基准相同。
4. S3 配额必须保持：`radar_rx` 4096/priority 5、`radar_local` 8192/priority 3、
   `radar_diag` 4096/priority 2、`radar_log` 6144/priority 1；任务栈、adapter workspace
   和 diagnostics snapshot 均使用 PSRAM，mutex 等同步控制对象维持静态存储，并逐项验证
   基准定义的失败清理。
5. 禁止修改 S3 的 internal RAM 配额或栈大小以“腾出空间”给雷达，除非先有 map、
   heap 与任务高水位证据。
6. 遥测限频、上传重试、日志队列必须有边界；离线时不允许积压无限 JSON 或阻塞
   network/voice worker。
7. 若保留 habit event reporter，单独验证其普通 3072 字节 internal task 栈、栈内 768 字节
   JSON 与 response buffer、创建失败、网络离线重试和 pending/drop 计数。该任务不属于
   基准雷达的 PSRAM 调度模型；未取得 map、heap、stack 高水位和离线重试证据前，不得将
   “整体资源调度正常”归因于基准雷达迁移。

构建门禁：按 C51、C52、S3 分开执行 IDF build，使用新建的隔离 build 目录，避免 Ninja
目录争用。构建成功仅证明编译、链接和静态资源布局，不能替代设备验收。

### 阶段 4：协议与调试工具兼容性

1. 抽取 C5 到 S3 的真实 `/local/v1/radar/result` 样本，验证 accepted、duplicate、
   identity mismatch、backward/conflict sequence 与 malformed target 的响应和日志。
2. 抽取实际 firmware 的 `RADAR_SOURCE_STATE event=counts`、`RADAR_ROOM_STATE`、
   `RADAR_HOME_STATE` 及空间状态日志，运行 ESPS3 Radar Debug Tool 的 parser checks 与
   SwiftPM build。`RADAR_COUNTS` 仅是 Debug Tool 的旧 fixture 兼容输入，不得把它当作
   当前固件的验收日志语法。
3. 确认 dashboard/日志消费者将 `source_person_count` 与 `home_person_count` 区分展示；
   不把短时 continuity track 描述成生物识别身份。

退出条件：解析器、工具和 firmware 日志字段对同一组样本得出相同的 source、room、HOME
人数和 presence/motion 结论。

### 阶段 5：硬件闭环验收

硬件验收必须在实际接线、固件版本和安装位置固定后完成。建议顺序：

1. 分开验证 C51/C52 的 BLE 绑定 MAC、address type、发现的 GATT notify 特征、订阅、帧率、
   断链和恢复；不得猜测 UUID 或把 UUID 当作 runtime handle。
2. 单独验证 S3 的 UART controller、GPIO TX/RX、`256000 8N1`、帧率、断线与 recovery；
   UART0 console 不得被本地雷达占用。
3. 空房、单人、两人、三人、进出、短暂遮挡/丢帧、雷达重启、C5 断连和 S3 重启逐项验证。
4. 验证 C51/C52/S3 同时在线时 HOME 是所有在线 source 的重算结果；依次离线每一源，
   确认旧状态按超时/连续性规则释放而不污染其他房间。
5. 语音、BME、Wi-Fi 重连和本地 HTTP 并发运行时连续观察至少一个业务周期，采集
   `STARTUP_MEMORY_CHECK`、heap、任务高水位、queue drop/retry、CPU/看门狗和网络错误。
6. 只有完成上述实机证据后，才可把“功能与基准完全一致、资源调度正常”写为硬件验收结论。

## 4. 验收矩阵

| 类别 | 通过标准 | 证据 |
| --- | --- | --- |
| 源码对等 | 所有预期目录一致；仅批准的 S3 下游扩展存在差异 | `diff -qr` 清单与代码审查 |
| C5 解析 | 分片、噪声、错误尾、三目标、坐标/速度解码均通过 | 两套 C5 host tests |
| S3 接入 | 大小、身份、序列、重复、离线与目标 schema 行为一致 | S3 radar-domain host tests |
| 资源 | 无启动 `ESP_ERR_NO_MEM`、无任务创建失败、可接受的 heap/stack 高水位 | build map、启动日志、设备运行日志 |
| 互操作 | C5 payload、S3 handler、日志和 Debug Tool 一致 | replay 样本与 parser checks |
| 真机 | 空房至三人、进出、干扰、断链恢复、三源聚合均通过 | 串口日志、Debug Tool 截图/记录、测试表 |

## 5. 明确禁止的“快速迁移”方式

- 不运行递归覆盖或复制整个项目目录；它会覆盖当前主项目的 habit HOME snapshot 及其他
  正在进行的非雷达工作。
- 不同时修改雷达算法、房间映射、BLE UUID、UART 引脚、任务栈和上传 schema；这些变量
  必须拆分，才能定位回归根因。
- 不用 build 成功代替雷达、BLE 或 UART 成功。
- 不在未取得硬件原始帧和设备日志时猜测 UUID、坐标、人与人之间的持续身份或安装参数。

## 6. 当前状态与下一步

根据 2026-07-20 的逐目录检查，阶段 0 的源码盘点已完成，且 C51/C52 与 S3 主链路
已经迁入当前主项目；当前剩余工作是保持差异受控，并按阶段 3-5 取得新的构建和实机证据。
本文件不执行代码迁移、构建、flash、monitor 或服务启动。
