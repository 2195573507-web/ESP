## 1. 本文范围与结论

本目录是一套围绕 **HLK-LD2450** 的完整开发资料包，覆盖模块选型、硬件接线、机械集成、UART 协议、蓝牙透传、Windows 调试、Android/iOS App 参考实现，以及 STM32 串口解析示例。

模块的核心能力是：在室内实时追踪最多 3 个运动目标，输出每个目标的横向坐标 `X`、纵向坐标 `Y`、速度和距离分辨率。对主控而言，最可靠的集成路径是：

1. 按 5 V / UART TTL 完成接线，默认 `256000, 8N1`。
2. 依据 `LD2450 串口通信协议 V1.03.pdf` 解析固定长度上报帧。
3. 对配置命令严格执行“进入配置模式 -> 等待 ACK -> 发送命令 -> 等待 ACK -> 结束配置模式”的事务流程。
4. 用 Windows 上位机或手机 App 完成现场验证；量产主控不要依赖 App 源码中其他型号的命令常量。

> 规范优先级：`LD2450 串口通信协议 V1.03.pdf` 是协议开发依据；`HLK LD2450 ... 说明书` 是硬件和安装依据；教程、参考代码和 App 工程只作为辅助示例。

## 3. 模块能力与关键指标

### 3.1 功能定位

LD2450 是 24 GHz ISM 频段的 FMCW 毫米波运动目标定位模组，使用一发两收微带天线与内置算法，面向家庭、办公室、酒店、照明、卫浴等室内人体运动定位场景。

它输出的是**运动目标**的位置和速度，不应把“可跟踪”理解为稳定的人体身份识别，也不应将其用作安全级人员存在判定。

### 3.2 已披露性能

| 项目 | 资料给出的值 |
| --- | --- |
| 工作频段 | 24 GHz - 24.25 GHz |
| 调制方式 | FMCW |
| 供电 | DC 5 V，电源能力大于 200 mA |
| 平均工作电流 | 120 mA |
| UART | 1 路 TTL UART，默认 256000 baud、8N1 |
| 最大跟踪数 | 3 个目标 |
| 最远跟踪距离 | 6 m |
| 方位角 | 约 +/-60 度 |
| 俯仰角 | 约 +/-35 度 |
| 数据刷新率 | 10 Hz |
| 扫频带宽 | 250 MHz |
| 工作温度 | -40 C 至 85 C |
| 模组尺寸 | 15 mm x 44 mm |

## 4. 硬件接线与结构集成

### 4.1 引脚和电平

| 引脚 | 作用 | 主控侧注意事项 |
| --- | --- | --- |
| `5V` | 模组电源输入 | 按大于 200 mA 的能力设计，控制纹波和毛刺 |
| `GND` | 电源地 | 必须与主控 UART 共地 |
| `Tx` | 模组串口发送 | 接主控 RX |
| `Rx` | 模组串口接收 | 接主控 TX |
| `PA9` | 有人高、无人低的 IO 输出 | 资料说明需通过 App 升级到 V2.05 才有；按 3.3 V 逻辑处理 |

模块同时提供插针与插座接口，两者均引出电源和串口；选择一个接口使用即可。教程中的 USB 转串口接线为 `5V-5V`、`GND-GND`、`TX-RX`、`RX-TX`。

### 4.2 安装要求

- 典型方式为挂壁安装，推荐安装高度 `1.5 m - 2 m`，天线正对探测区域。
- 天线前方和周边应开阔，模组本体必须固定，避免晃动、挤压和变形。
- 避开持续运动的窗帘、风口绿植、风扇、空调、强反射平面和多个正对安装的 24 GHz 雷达。
- 模组后方的运动也可能被背瓣感知；需要时可用金属背板/屏蔽罩减弱后向干扰。金属不得位于天线前方。
- 多个雷达应错开波束方向并拉开距离。通用天线罩指南建议模块之间至少 `0.5 m`。

### 4.3 天线罩与外壳

- 天线罩表面要平整、均匀，天线正上方应与天线面平行；涂层不能含金属或导电材料。
- 24.125 GHz 下，空气半波长约 `6.2 mm`。资料建议天线到天线罩内壁的间隙优先取 `12.4 mm` 或 `18.6 mm`，误差控制在 `+/-1.2 mm`。
- 天线罩厚度优先取材料中的半波长，容差约 `+/-20%`；无法满足时应优先低介电常数材料，并考虑 `1/8` 波长或更薄。
- 示例材料：ABS（相对介电常数约 2.5）半波长约 `3.92 mm`；PC 约 `3.58 mm`；HDPE 约 `4.00 mm`。实际材料参数必须向供应商确认。
- 资料还给出更保守的通用建议：天线面与产品外壳保持 `3-5 mm`。若与前述波长优化间隙冲突，应以实测覆盖范围、误报率和结构约束综合定版。

## 5. UART 数据上报协议

### 5.1 帧结构

默认串口为 `256000, 8N1`，采用小端字节序，每秒上报 10 帧。

```text
AA FF 03 00 | target_1 (8 bytes) | target_2 (8 bytes) | target_3 (8 bytes) | 55 CC
```

- 总长度：`4 + 3 * 8 + 2 = 30 bytes`。
- 无目标槽位的 8 字节均为 `00`。
- 每个目标按以下 4 个 little-endian 16-bit 字段连续排列：`X`、`Y`、`speed`、`resolution`。

### 5.2 坐标与速度的特殊编码

这不是普通的二补码 `int16_t` 解释。资料定义为“最高位表示方向，剩余 15 位为幅值”：

| 字段 | 原始值 | 计算方式 | 单位 |
| --- | --- | --- | --- |
| `X` | `raw = lo + hi * 256` | `hi & 0x80` 时 `raw - 32768`，否则 `-raw` | mm |
| `Y` | `raw = lo + hi * 256` | `raw - 32768`（有效目标通常最高位为 1） | mm |
| `speed` | `raw = lo + hi * 256` | `hi & 0x80` 时 `raw - 32768`，否则 `-raw` | cm/s |
| `resolution` | `lo + hi * 256` | 直接使用 | mm |

协议示例中的第一个目标 `0E 03 B1 86 10 00 40 01` 解析为：

```text
X = -(0x030E) = -782 mm
Y =  0x86B1 - 32768 = 1713 mm
speed = -(0x0010) = -16 cm/s
resolution = 0x0140 = 320 mm
```

如果需要显示极坐标，可在主控层计算：

```text
distance_mm = sqrt(X * X + Y * Y)
angle_deg  = atan2(X, Y) * 180 / pi
```

角度的正负方向必须与 UI 坐标系统一；参考 Kotlin 代码为适配其显示视图做了额外旋转/取反，不能不加验证地照搬。

### 5.3 建议的生产解析器

1. 以字节流方式累积 UART 数据，搜索帧头 `AA FF 03 00`。
2. 收满 30 字节后验证末尾 `55 CC`，再解析 3 个目标槽位。
3. 帧头或帧尾不匹配时逐字节重新同步，而不是整体清空串口缓冲。
4. 仅将完整、校验通过的帧发布给上层；连续异常帧应计数并上报诊断。
5. 对全零的目标槽位显式标为“不存在”，不可将其解释成坐标原点的真实目标。

STM32 参考工程实现了 `AA FF 03 00`、24 字节载荷、`55 CC` 的接收状态机，可用作结构参考。单页 C 示例只检查帧头、边界检查不足且未验证完整帧尾，故不适合直接投产。

## 6. UART 配置命令与 ACK

### 6.1 通用命令格式

```text
发送：FDFC FBFA | payload_length (uint16 LE) | command (uint16 LE) | value | 04030201
ACK ：FDFC FBFA | payload_length (uint16 LE) | (command | 0x0100) | result/value | 04030201
```

`result` 为 `0000` 表示成功，`0001` 表示失败。配置时不要依据“已发送”判定成功，必须等待对应 ACK。

### 6.2 配置事务

```text
主控                         LD2450
 |-- enable config (0x00FF) -->|
 |<--------- ACK success ------|
 |-- target configuration ---->|
 |<--------- ACK success ------|
 |-- end config (0x00FE) ---->|
 |<--------- ACK success ------|
```

每次新的配置会话都必须重新使能。进入配置模式后数据上报可能暂停，结束配置后才恢复工作状态。

### 6.3 LD2450 命令速查

| 功能 | 命令字 | 发送帧（空格仅为阅读） | 说明 |
| --- | ---: | --- | --- |
| 进入配置 | `0x00FF` | `FDFCFBFA 0400 FF00 0100 04030201` | 其他配置前必须发送 |
| 结束配置 | `0x00FE` | `FDFCFBFA 0200 FE00 04030201` | 恢复工作模式 |
| 单目标跟踪 | `0x0080` | `FDFCFBFA 0200 8000 04030201` | 设置单目标 |
| 多目标跟踪 | `0x0090` | `FDFCFBFA 0200 9000 04030201` | 设置多目标，出厂默认 |
| 查询跟踪模式 | `0x0091` | `FDFCFBFA 0200 9100 04030201` | 返回 `0100` 单目标或 `0200` 多目标 |
| 查询固件版本 | `0x00A0` | `FDFCFBFA 0200 A000 04030201` | 返回类型、主版本和次版本 |
| 设波特率 | `0x00A1` | 见下一表 | 掉电保存，重启后生效 |
| 恢复出厂 | `0x00A2` | `FDFCFBFA 0200 A200 04030201` | 重启后生效 |
| 重启 | `0x00A3` | `FDFCFBFA 0200 A300 04030201` | ACK 发出后重启 |
| 蓝牙开 | `0x00A4` | `FDFCFBFA 0400 A400 0100 04030201` | 掉电保存，重启后生效 |
| 蓝牙关 | `0x00A4` | `FDFCFBFA 0400 A400 0000 04030201` | 掉电保存，重启后生效 |
| 查询 MAC | `0x00A5` | `FDFCFBFA 0400 A500 0100 04030201` | 返回 3 字节大端 MAC 段 |
| 查询区域过滤 | `0x00C1` | `FDFCFBFA 0200 C100 04030201` | 返回模式和 3 个矩形区域 |
| 设置区域过滤 | `0x00C2` | `FDFCFBFA 1C00 C200 ... 04030201` | 立即生效且掉电保存 |

波特率索引：`0001=9600`、`0002=19200`、`0003=38400`、`0004=57600`、`0005=115200`、`0006=230400`、`0007=256000`、`0008=460800`。例如恢复默认速率对应的设置帧是 `FDFCFBFA 0400 A100 0700 04030201`。

### 6.4 区域过滤

区域过滤类型为：`0000` 关闭、`0100` 仅检测设定区域、`0200` 排除设定区域。支持最多 3 个矩形区域，每个区域由两组对角点 `(x1, y1, x2, y2)` 组成，坐标为带符号 16 位、单位 mm；全零区域表示未使用。

“仅检测”和“排除”不是可同时叠加的两种功能。教程也明确指出二者不能同时生效。

## 7. 蓝牙与透明传输

### 7.1 普通 App 工程中使用的 BLE 服务

Android 示例的 `BluetoothLeClass.java` 使用以下 GATT 标识：

| 项目 | UUID |
| --- | --- |
| 服务 | `0000fff0-0000-1000-8000-00805f9b34fb` |
| 通知/读取特征 | `0000fff1-0000-1000-8000-00805f9b34fb` |
| 写入特征 | `0000fff2-0000-1000-8000-00805f9b34fb` |

Android 工程请求 MTU `247`，订阅通知并缓存分包数据；iOS 工程使用 BabyBluetooth，在写特征上进行 `CBCharacteristicWriteWithoutResponse`。这说明两端示例都按 BLE 串口式数据通道组织，而不是按单个业务特征拆分。

### 7.2 固件 2.14.25112412 的透明传输文档

该独立 PDF 定义的 UUID 与上面的 App 工程不同：

| 项目 | UUID |
| --- | --- |
| 蓝牙透传服务 | `0000fff0-0000-1000-8000-00805f9b34fb` |
| 手机写入 | `0000fff3-0000-1000-8000-00805f9b34fb` |
| 手机读取 | `0000fff4-0000-1000-8000-00805f9b34fb` |

透明传输还要求应用层封包：

```text
BLE -> UART：AA FF 55 00 | custom data | 55 CC
UART -> BLE：AA FF 55 01 | custom data | 55 CC
```

因此，连接前必须根据实际固件确认特征 UUID，不能假定旧 App 的 `FFF1/FFF2` 与新固件的 `FFF3/FFF4` 可互换。

## 8. 软件与工具资料说明

### 8.1 Windows 首次联调

1. 根据 USB 转串口板芯片安装 `CP210x` 或 `CH341` 驱动。
2. 连接模块并打开对应 COM 口。
3. 先以 `256000, 8N1` 观察是否持续收到以 `AA FF 03 00` 开头、`55 CC` 结尾的 30 字节帧。
4. 使用 `HLK-2450_TOOL.exe` 的检测功能搜索设备，再开始显示目标扇形图、距离、角度和速度。
5. 若需下发配置，先发送进入配置帧并检查 ACK；不要一边持续配置一边假定上报不会中断。

### 8.2 Android 参考工程

- 工程应用 ID：`com.hlk.hlkradartooldemo`。
- 构建配置：`compileSdk 33`、`minSdk 26`、`targetSdk 34`、Java/Kotlin 1.8。
- 依赖包含 AndroidX、Material、EventBus、Gson 和项目内 BLE library。
- `BaseVolume.kt` 包含明确标记 `_2450` 的命令常量，可与 V1.03 协议交叉核对。
- 同一工程还含 LD2410、LD2411、LD2412、LD2451、LD6002 等型号代码。解析或复制常量前应按型号筛选，不能以文件中存在某命令就认定 LD2450 支持它。

### 8.3 iOS 参考工程

- iOS 最低平台版本为 `11.0`。
- Podfile 使用 `IQKeyboardManager`、`Masonry`、`BabyBluetooth ~> 0.7.0`。
- 工程有独立的扫描/连接控制器和 `ReceiveSendViewController`，可用于参考 GATT 写入与通知接收，但其第三方依赖版本较老，建议新项目用当前维护的 BLE 封装重写传输层。

### 8.4 STM32 示例工程

- Keil uVision 工程目标：`STM32F103C8`、Cortex-M3、ARMCC 5.06。
- `Serial.c` 将 USART1 配为 `256000, 8N1`，中断状态机识别帧头、收取 24 字节目标载荷并校验帧尾。
- 这是可移植的参考结构，但工程使用全局状态、在接收中断中逐步解析字段，主项目应考虑将完整帧交给任务/队列解析，降低中断负担并便于做超时和统计。

| 风险 | 证据 | 建议 |

| 规格书版本标记不一致 | 文件名和修订记录为 V1.02，封面仍写 V1.01 | 以修订记录和实际固件查询结果为准，发布前向供应商确认 |

| App 工程是多型号集合 | Android 常量含多种 LD24xx/LD6002 命令 | LD2450 仅采用协议 V1.03 已定义命令 |
| 目标字段是专用符号编码 | 文档以最高位定义方向，并非直读二补码 | 封装专用解码函数，并用文档样例做单元测试 |
| 单页 C 样例不完整 | 未验证完整帧尾，且边界判断与实际访问长度不严谨 | 使用 30 字节状态机和完整帧验证 |
| 老旧 Windows 二进制与驱动 | ZIP/RAR 内含 exe/dll | 在隔离 Windows 环境、校验来源后运行 |
| 部分 ZIP 文件名为旧编码 | macOS `unzip` 会出现非法字节路径错误 | 保留原包；需要解压时在 Windows 或支持 GBK/CP936 的工具中处理 |

## 10. 推荐集成检查表

- [ ] 供电为稳定 5 V，电流能力至少 200 mA，并检查纹波、地线和开关电源距离。
- [ ] UART 交叉连接并配置为 `256000, 8N1`；主控 IO 电平与模块接口兼容。
- [ ] 首帧验证通过：`AA FF 03 00` + 24 字节载荷 + `55 CC`，总长 30 字节。
- [ ] 用协议示例验证 `X/Y/speed/resolution` 的符号和单位换算。
- [ ] 配置前后均等待 ACK；处理超时、失败 ACK 和模块重启后的重连。
- [ ] 查询并记录固件版本、追踪模式、波特率、蓝牙状态和区域过滤配置。
- [ ] 通过上位机或 App 在实际安装位置验证覆盖、盲区、窗帘/风扇/背瓣干扰。
- [ ] 外壳、天线罩和雷达间距通过样机实测定版，不以理论波长值替代测试。
- [ ] 若使用 BLE，按实际固件发现 UUID 并做长包分片/重组与断线恢复。

## 11. 资料阅读顺序

1. 先读 `HLK LD2450 ... 说明书`：完成选型、供电、接线和安装。
2. 再读 `LD2450 串口通信协议 V1.03`：实现数据帧和配置状态机。
3. 参考 `HLK-LD2450使用教程V1.1`：完成 Windows/手机首次通电验证。
4. 参考 STM32 和 Kotlin 解析包：校对字节序、坐标和 UI 角度计算。
5. 需要 BLE 透明通道时，最后阅读 `2450透传使用说明` 并先确认模块固件版本。

## 12. 附：最小主控伪代码

```c
// 串口字节流状态机：只在获得完整 30 字节帧后发布结果。
if (frame[0] == 0xAA && frame[1] == 0xFF &&
    frame[2] == 0x03 && frame[3] == 0x00 &&
    frame[28] == 0x55 && frame[29] == 0xCC) {
    for (int target = 0; target < 3; ++target) {
        const uint8_t *p = &frame[4 + target * 8];
        uint16_t x_raw = p[0] | ((uint16_t)p[1] << 8);
        uint16_t y_raw = p[2] | ((uint16_t)p[3] << 8);
        uint16_t v_raw = p[4] | ((uint16_t)p[5] << 8);
        uint16_t res   = p[6] | ((uint16_t)p[7] << 8);

        if (x_raw == 0 && y_raw == 0 && v_raw == 0 && res == 0) continue;

        int16_t x_mm = (p[1] & 0x80) ? (int16_t)(x_raw - 32768) : -(int16_t)x_raw;
        int16_t y_mm = (int16_t)(y_raw - 32768);
        int16_t v_cms = (p[5] & 0x80) ? (int16_t)(v_raw - 32768) : -(int16_t)v_raw;
        publish_target(target, x_mm, y_mm, v_cms, res);
    }
}
```

上述伪代码仅对应 V1.03 上报帧。实际工程还应加入环形缓冲、帧超时、统计计数、线程/中断边界和固件版本识别。
