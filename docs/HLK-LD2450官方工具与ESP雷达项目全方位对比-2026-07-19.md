# HLK-LD2450 官方工具与 ESP 雷达项目全方位对比

日期：2026-07-19  
对比对象：`/Users/zhiqin/Documents/ESP 资料` 中的 HLK-LD2450 官方资料、示例工程、Android/iOS 工具源码、Windows 工具包，以及当前项目 `/Users/zhiqin/ESP 雷达`。

## 1. 结论先行

### 1.1 两者不是同一层的产品

HLK 官方工具主要解决三件事：

1. 连接雷达、订阅/透传数据和发送配置命令；
2. 解码当前雷达帧，显示三个目标的坐标、速度、距离、角度等派生量；
3. 让工程人员配置单目标/多目标模式、区域过滤、蓝牙和模块参数。

当前 ESP 雷达项目解决的是更长的链路：

```text
LD2450 UART 或 C5 BLE
  -> 固定帧解析/重同步
  -> 原始目标保留
  -> 坐标变换和空间准入
  -> 跨帧目标关联与 track_id
  -> 滤波、置信度、失帧保持
  -> 区域、占用、运动、传感器健康状态
  -> C5-S3 身份/序列保护
  -> 本地注册表、HTTP/JSON、日志和 macOS 回放可视化
```

所以，官方工具“看起来能显示目标”与 ESP 项目“能输出稳定人体轨迹/人数状态”不是同一个能力层。官方协议负责输出最多三个当前目标槽；ESP 项目在其上增加了业务层时序和空间语义。

### 1.2 最重要的判断

| 判断 | 结论 | 证据等级 |
|---|---|---|
| LD2450 是否直接输出持久人体 ID | 否。实时协议只有三个目标槽，没有跨帧持久 ID 字段 | A：协议 + 官方 STM32 示例 |
| 官方工具是否必然实现了 EMA/Kalman/最近邻等轨迹算法 | 不能证明。可读 Android/STM32 代码没有这类跨帧算法；Windows 工具是闭源二进制 | A/B |
| 官方工具能否显示三个目标 | 能。它解码三个 8 字节槽，协议和官方示例均明确最多三个目标 | A |
| 官方工具能否改变目标模式和区域过滤 | 能。Android 源码包含单/多目标、三矩形过滤的查询/设置命令 | A |
| ESP 项目能否生成稳定的业务 track_id | 软件上能。S3 tracker 通过跨帧匹配、确认帧、置信度和生命周期生成自有 ID | A：当前源码 |
| ESP 项目的“人数”是否等于官方 UI 的人数 | 不能直接等同，必须明确比较层：raw、accepted、visible、confirmed active track | A/B |
| 当前实现是否已经完成真机等价验证 | 没有。已有 host tests/build 只能证明软件逻辑；BLE GATT、安装标定、现场精度仍需硬件闭环 | D |

### 1.3 总体评价

- **官方工具的优势**：直接贴合模块固件和配置协议，适合首次联调、读回参数、切换目标模式、画当前目标、检查 BLE/UART 通道。
- **官方工具的限制**：工具源码/协议没有证明稳定的人体身份跟踪、跨帧人数去抖、失联状态机、房间语义或多雷达融合。显示稳定可能来自模块固件、当前帧替换、显示节流或闭源逻辑，不能猜算法。
- **ESP 项目的优势**：把原始目标和业务目标分层，具备 track ID、EMA、速度/跳变拒绝、空间区域、占用/运动滞回、UART 恢复、BLE 身份与序列校验、日志回放和多源汇合。
- **ESP 项目的风险**：参数比官方工具复杂；如果拿 `target_count`、槽号、accepted count、track count 混为一谈，或没有先对齐雷达的单/多目标与区域配置，比较结果会失真。当前真实硬件验证仍不完整。

### 2.2 证据等级

- **A：源码/协议直接证明**。例如帧头、帧长、UUID 常量、命令、当前项目的函数和字段。
- **B：工具源码或配置证明可见能力**。例如 Android 有区域配置界面/命令流程，但不能证明闭源 Windows 版本使用同一实现。
- **C：基于字段和结构的静态推断**。必须写成“可能/应当”，不能写成官方事实。
- **D：需要真机验证**。包括 BLE 地址类型、实际 GATT、模块固件版本、安装轴向、多径、现场人数准确率和官方工具与 ESP 同帧对照。

Android 源码也写死了一个 LD2450 BLE 配置分支：

- Service：`0000ae00-0000-1000-8000-00805f9b34fb`
- Notify/Read：`0000ae02-0000-1000-8000-00805f9b34fb`
- Write：`0000ae01-0000-1000-8000-00805f9b34fb`

这里应理解为“官方 Android 源码中的历史/特定型号绑定证据”，不是可以直接复制到 ESPC51/C52 的生产结论。不同固件、透传版本、地址类型和设备批次必须实机 discovery；当前 ESP 项目仍遵守 fail-closed 绑定原则。

### 3.3 iOS 蓝牙 Demo

iOS Demo 的价值主要在 BLE 扫描、连接、通知订阅和命令收发参考。它属于工具/联调层，不应拿来证明 LD2450 内部的多目标关联算法。与 Android 一样，真正可复用的应是“发现服务 -> 找通知特征 -> 订阅 -> 收到字节 -> 按设备状态解析”的流程，而不是把某个 UUID 或显示层模型未经验证地移植成固件合同。

### 3.4 STM32 官方解析工程

这是本次最有价值的官方可读算法参考，文件：

- `Hardware/Serial.c`
- `User/main.c`

其行为非常明确：

1. USART1 配置 `256000, 8N1`；
2. 中断状态机寻找 `AA FF 03 00`；
3. 接收 24 字节 payload；
4. 检查 `55 CC` 帧尾；
5. 对三个槽分别解码 X、Y、Speed、DistanceResolution；
6. 用 `sqrt(x*x+y*y)` 算距离，用 `-atan2(x,y)` 算角度；
7. 打印三个槽的即时结果。

工程没有实现：

- 跨帧人物 ID；
- 最近邻/全局 assignment；
- EMA、Kalman 或其他平滑；
- 速度预测；
- 失帧保持和轨迹过期；
- 房间/区域业务状态；
- “静止人体”与“运动人体”状态机；
- 人体类别或人体置信度。

这不是说官方模块内部一定没有目标跟踪，而是说“主控侧官方示例”没有再次实现这些能力。

## 4. 数据格式与解析对比

### 4.1 LD2450 官方实时帧

官方协议和示例对应的实时帧为固定 30 字节：

```text
AA FF 03 00                         帧头 4 字节
slot 1: X(2) Y(2) Speed(2) Res(2)  8 字节
slot 2: X(2) Y(2) Speed(2) Res(2)  8 字节
slot 3: X(2) Y(2) Speed(2) Res(2)  8 字节
55 CC                               帧尾 2 字节
```

字段为小端序。X、Speed 使用方向编码，Y 使用带偏移的有符号编码。全零槽表示没有可显示目标；协议没有 `target_id`、CRC 或人体分类字段。`DistanceResolution` 是模块上报字段，主控另行计算欧氏距离只是派生值，二者不能混名。

### 4.2 官方 STM32 与 ESP S3 parser

| 项目 | 官方 STM32 示例 | 当前 ESP S3 |
|---|---|---|
| 接收方式 | UART RX 中断状态机 | UART 服务任务独占读取，parser 独立处理字节流 |
| 帧边界 | 找头、收 24 字节、查尾 | 同样的 30 字节合同，逐字节重同步 |
| 粘包/分片 | 示例没有完整处理说明 | 支持半帧保留、粘包和错误后滑动重同步 |
| 坏头/坏尾 | 简单丢弃/回到状态 0 | 记录 bad header、bad tail、skipped/discarded、resync |
| 目标槽 | 3 个固定结构体 | 3 个 raw target，带 slot、valid、时间、frame_seq |
| 派生距离 | `double sqrt` | 整数平方根，输出 `distance_mm` |
| 无效值 | 主要检查槽是否为零 | 全零、零分辨率、sentinel、范围、房间准入分层处理 |
| 诊断 | 基本没有 | 帧率、最大帧间隔、半帧长度、超时、恢复状态等 |
| 输出语义 | 立即打印当前槽 | raw frame -> accepted target -> track/spatial snapshot |

### 4.3 ESP C5 BLE parser 与 S3 parser 的分工

ESPC51/C52 的 BLE 链路先接收通知字节并解析 LD2450 帧，做基础合法性和资源/传输控制；S3 负责空间语义和跨帧 track。这样做的好处是：C5 不承担房间坐标和人体业务计算，S3 可以统一处理本地 UART、C5-1、C5-2 三类来源。

C5 上送的是结构化结果，不是把 raw BLE 帧直接交给 Server。结果包含：

- schema/version；
- 固定 `local_id` 和设备身份；
- request/frame sequence、uptime；
- link/sample validity；
- target_count 和目标数组；
- x/y/speed/resolution/distance/confidence。

S3 对远端输入还检查 JSON 精确字段、数组长度、slot 唯一性、序列、会话身份和来源 IP。这些是官方工具没有覆盖的系统级可靠性能力。

## 6. 人体轨迹和目标身份对比

### 6.1 官方工具能证明什么

官方协议的三个槽是“本帧位置槽”。官方 STM32 示例变量名是 `m1/m2/m3`，Android 的可见坐标处理也是按当前上报数据和矩形配置计算。源码没有发现：

- `track_id` 的跨帧分配；
- 依据上一帧位置的匹配；
- 轨迹确认帧数；
- 轨迹失联保持；
- 交叉时的身份交换处理；
- 轨迹删除和历史轨迹保存。

因此不能把 `slot 1` 当成“同一个人 1”。多人交叉、槽位重排、暂时丢失时，槽号可能仍是槽号而不是人物身份。

### 6.2 当前 ESP S3 tracker 的实际实现

当前实现位于：

- `ESPS3/components/Middlewares/radar_domain/radar_target_tracker.c`
- `ESPS3/components/Middlewares/radar_domain/radar_spatial_state.c`
- `ESPS3/components/Middlewares/radar_domain/include/radar_spatial_types.h`

它的行为是：

1. 最多维护 `LD2450_MAX_TARGETS` 条轨迹，即 3 条；
2. 用上一位置/速度预测位置，再做目标分配搜索；
3. 以距离门限和速度异常检查拒绝明显不合理匹配；
4. 新观察先进入 `TENTATIVE`，达到确认帧数和置信度后才拥有单调递增 `track_id`；
5. 命中后保存 raw 坐标、filtered 坐标、速度、方向、位移、置信度、zone、首次/最后出现时间；
6. 未命中时递减置信度，短期进入 `HOLD`，超过超时释放；
7. 当前输出排除 tentative/hold，历史快照单独保存，不把过期轨迹混入当前人数。

当前配置入口 `ESPS3/components/radar_ld2450/include/radar_config.h` 的实时值为：

| 参数 | 当前值 | 含义 |
|---|---:|---|
| 关联门限 | 900 mm | 目标与预测位置的匹配范围；另有 800/1200 mm 配置边界 |
| 轨迹确认 | 2 帧 | 连续观察后才生成稳定 ID |
| EMA alpha | 60% | 当前测量对 filtered 坐标的权重 |
| 速度异常上限 | 3000 mm/s | 位置推导速度的保护阈值 |
| HOLD | 500 ms | 短暂失帧仍保留历史轨迹生命周期 |
| 轨迹释放 | 1800 ms | 超时后从活动 tracker 中删除 |
| 轨迹最低置信度 | 60 | 低于该值不作为当前稳定轨迹 |
| 运动速度阈值 | 15 cm/s | 运动候选条件之一 |
| 运动位移阈值 | 120 mm | 运动候选条件之二 |
| 运动进入/退出 | 3/8 帧 | 运动滞回，减少抖动 |
| 目标突跳 | 1500 mm | 单目标空间突跳保护 |

这些是 ESP 自己的工程参数，不是官方协议字段，也不能声称是官方 App 参数。

### 6.3 轨迹能力结论

- **官方模块固件**：已经负责雷达回波检测和目标输出，且协议/模式名称使用“追踪”；但协议不把内部轨迹身份暴露给主控。
- **官方主控示例**：只演示当前帧槽解码和显示。
- **ESP 项目**：显式实现主控侧跨帧关联和业务轨迹，能对外提供自有 `track_id`，但其身份稳定性必须用多人交叉、进出场和丢帧回放及真机测试验证。

## 7. 人体个数判断对比

### 7.1 官方侧的“个数”

官方实时帧最多三个槽，最直接的数量是：

```text
当前帧中非空/有效的目标槽数量
```

Android 工具还可以切换单目标/多目标模式；如果模块处于单目标模式，不能拿工具显示的 1 推导“房间里一定只有 1 个人”。区域过滤也会改变模块实际输出，配置为“仅区域内”时区域外目标不会进入当前帧。

官方工具的数量至少可能有三种含义：

1. 当前帧非零槽数量；
2. 模块过滤/模式后的输出数量；
3. UI 为了显示稳定而暂时保留的数量。

闭源 Windows 工具没有源码证据时，不能确定它采用哪一种后处理。

### 7.2 ESP 侧必须分层比较

当前 S3 快照明确区分：

| 数量 | 含义 | 用途 |
|---|---|---|
| `raw_target_count` | 完整 LD2450 帧中解析出的有效槽数量 | 协议/串口诊断 |
| `accepted_target_count` | 通过零分辨率、sentinel、距离、房间边界和坐标准入的数量 | 空间算法输入 |
| `visible` track count | 当前帧可见的轨迹数量 | 当前画面/当前状态 |
| `confirmed active_track_count` | 已确认且达到置信度门槛的活动轨迹数量 | 业务人数、registry 输出 |
| `history_target_count` | 已离开当前视野但保留用于回放/诊断的历史轨迹数量 | 不能计入当前人数 |

正确的业务人数应优先使用 `confirmed active_track_count`，而不是 raw 槽数量或 history 数量；但“传感器离线”时不得把人数强行改成 0，而应输出 UNKNOWN/不可确认。

### 7.3 人数准确性边界

软件上，ESP tracker 将人数从“每帧槽数”提升为“稳定活动 track 数”。但 LD2450 本身最多三目标，且协议没有人体类别和持久 ID，所以以下场景必须真机验收：

- 两人靠近或交叉；
- 一人短时被遮挡/丢失；
- 一人离场、另一人进入；
- 静止人体与风扇/窗帘/金属反射共存；
- 区域边界来回移动；
- 单目标/多目标模式切换后的人数稳定性。

## 8. 运动、存在和静止人体判断

| 维度 | 官方工具/示例 | ESP 项目 |
|---|---|---|
| 运动速度 | 显示每个槽的 speed | 使用 speed + 帧间位移判断运动候选 |
| 有人/无人 | LD2450 模组资料可用于运动目标输出，但 STM32 示例没有业务状态机 | `PRESENT`、`HOLD`、`VACANT_INFERRED`、`UNKNOWN` |
| 静止人体 | 官方 LD2450 的基础定位不是主控静止人体业务状态证明 | `STILL_CANDIDATE` 与存在状态分离 |
| 进入/退出抖动 | 示例无滞回 | 运动 3 帧进入、8 帧退出 |
| 离线 | 示例无统一 sensor state | `VALID`/`STALE`/`OFFLINE`，离线不伪装 vacant |
| 置信度 | 官方协议无人体 confidence 字段 | ESP track quality score，不能解释为雷达原生置信度 |

特别注意：ESP 的 `confidence` 是软件轨迹质量分数，官方 wire frame 没有这个人体置信度字段。它只能回答“这个跨帧 track 当前有多稳定”，不能回答“雷达已经识别为人且概率为多少”。

## 9. 区域过滤与空间语义对比

### 9.1 官方模块级区域过滤

Android 源码明确支持最多三组矩形坐标，并有三种模式：

- 关闭过滤；
- 仅检测设置区域；
- 不检测设置区域。

这是模块配置层能力，目的是让雷达固件减少某些空间区域的目标输出。官方工具还负责读取、编辑、换算视图坐标并发出 `C200` 设置命令。

### 9.2 ESP S3 空间层

ESP 使用 `radar_coordinate_transform` 和 `radar_zone_map`：

- 镜像、旋转、原点平移；
- 最大探测距离和房间矩形准入；
- active/entry/ignore 等业务区域；
- 上一 zone 的滞回；
- zone changed、zone left、entered time、dwell time；
- 空间准入前保留 raw，准入后另存 accepted。

当前默认坐标不翻转、不旋转，房间范围约为 X/Y `-6000..6000 mm`，active 区域覆盖默认房间并使用 `250 mm` 区域滞回。这是保守的软件配置，不等于已经完成安装标定。

### 9.3 两层区域不能混淆

官方矩形过滤改变雷达模块“是否上报”；ESP zone map 改变主控“如何解释、计数和发布”。推荐顺序是：

```text
模块级过滤：先减少明确的固定干扰
  -> ESP 原始帧保留：记录真实输入
  -> 坐标校准/房间准入
  -> 业务 zone：判断房间、入口、忽略区、停留时间
```

调试阶段应避免同时随意修改两层，否则无法知道人数差异来自雷达固件过滤、主控空间准入，还是轨迹状态机。

## 10. BLE、身份和多源传输对比

### 10.1 官方 Android BLE 流程

源码流程是：扫描设备 -> 连接 GATT -> 服务发现 -> 匹配 Service/Read/Write UUID -> 写 CCCD 订阅通知 -> 收到字节 -> 按设备 MAC 和型号解析 -> 回调 UI。它还会对连接失败进行有限次重连。

这证明了官方工具需要完整的 GATT 合同，但不证明所有 LD2450 固件都使用同一 Service/Notify/Write UUID。

### 10.2 ESP C5 BLE 流程

当前 C5 设计比通用工具更严格：

- C51/C52 绑定固定设备身份；
- 不自动连接附近同名设备；
- Service/Notify UUID 未验证时保持禁用；
- 通知收到后进入 bounded/latest-only 数据路径；
- C5 做采集和基础解析，S3 做空间计算；
- 上传包含 local_id、device_id、sequence、uptime、链路/样本有效性和目标数组。

优点是不会把“附近同名雷达”错当成目标设备；代价是必须先拿到真实雷达的地址类型、MAC、Service UUID、Notify UUID，必要时还要拿到 Write UUID。

### 10.3 C5-S3 与官方工具的职责差异

| 项目 | 官方工具 | ESP C5-S3 |
|---|---|---|
| 来源身份 | 手机/电脑当前选中的 BLE MAC | 固定 local_id + device_id + session/peer 校验 |
| 数据传输 | BLE 直接给 UI | C5 采集 -> S3 结果协议 -> S3 空间层 |
| 缓存 | 工具内设备状态缓存 | bounded buffer/latest-only，避免阻塞音频/网络 |
| 顺序 | 工具回调顺序为主 | sequence/duplicate/backward/identity 校验 |
| 多雷达 | 手工选择设备 | S3 为 local、C51、C52 分开维护 slot |
| 业务状态 | UI 内部显示 | registry、HTTP、日志、占用和运动状态 |

## 11. 容错、恢复和可观测性

### 11.1 官方示例的边界

STM32 示例是教学/验证代码，接收中断里就开始写入和解析字段，完整帧后设置标志。它没有完整的：

- UART 驱动重启；
- 静默检测和指数退避；
- 半帧老化策略；
- 原始字节诊断环形缓冲；
- 丢帧与解析错误分类；
- 设备离线与业务 vacant 分离。

这不代表官方 Windows/App 一定没有这些逻辑，只代表随附可读示例没有证明它们。

### 11.2 ESP S3 当前能力

- parser 对坏头/坏尾逐字节重同步；
- 保存半帧并记录 partial timeout；
- UART 读错误/静默进入 recovery/backoff；
- 恢复后要求连续有效帧再回到 VALID；
- latest frame 与空间 snapshot 分离；
- raw HEX 调试默认关闭，必要时按上限和时间间隔输出；
- 诊断包括字节数、有效帧、帧率、最大间隔、丢弃、重同步、恢复状态；
- S3 远端输入校验精确 JSON schema、序列、身份、会话和 peer IP。

因此 ESP 的“稳定”不仅是画面稳定，也包括链路故障时不制造假人数、不接受错源数据和能解释为什么丢帧。

## 12. 可视化和调试工具对比

### 12.1 官方工具

官方 Windows/Android/iOS 工具适合：

- 看当前三个目标；
- 查看距离、角度、速度等派生量；
- 读取/修改模块配置；
- 现场判断 UART/BLE 是否通；
- 验证区域过滤和单/多目标模式。

### 12.2 ESP macOS Debugger

`ESPS3-Radar-Debug` 是独立的工程调试工具，能：

- 读取 ESPS3 控制台日志；
- 导入和回放日志；
- 区分 raw slot、accepted target、track；
- 显示 track ID、raw/filtered 坐标、状态、诊断；
- 查看多源和 UNKNOWN 记录；
- 调整房间/区域显示配置并清理轨迹。

它的轨迹画布不是雷达传感器算法，不能用画布上的线条反推官方模块内部是否有稳定身份。

## 14. 推荐的同场对比方法

要真正回答“官方工具和 ESP 项目谁的轨迹/人数更准”，必须做同源数据对比，而不是凭两块屏幕目测：

### 阶段 A：先冻结配置

记录：

- LD2450 固件版本；
- 单目标/多目标模式；
- 模块区域过滤模式和三组矩形；
- UART 波特率（默认 256000, 8N1）；
- 模块安装高度、朝向、天线罩和房间边界；
- 官方工具版本；
- ESP 固件 build/配置版本。

### 阶段 B：同步采集四类证据

同一时间记录：

1. 官方工具原始/显示结果；
2. ESP `RADAR_RAW_FRAME` 或受控 UART 原始字节；
3. ESP parser 的 `raw_target_count`、错误和 `frame_seq`；
4. ESP spatial 的 `accepted_target_count`、track ID、visible/confirmed count、occupancy/motion。

### 阶段 C：按场景回放

至少测试：

- 空房间；
- 一人静止；
- 一人走近/走远；
- 两人并行；
- 两人交叉；
- 一人短时离开探测区；
- 区域边界进出；
- 风扇/窗帘/空调等干扰；
- UART 拔插、噪声、BLE 断连/重连。

### 阶段 D：使用明确指标

- 帧有效率、最大帧间隔、重同步次数；
- raw/accepted/confirmed 三种数量的差异；
- track ID 切换次数；
- 误增人数、漏减人数、短暂 ghost track 数；
- 进入/退出延迟；
- 静止人体误判运动次数；
- 离线期间是否错误输出 vacant；
- zone switch、zone dwell 和边界抖动次数。

只有同一原始帧、同一配置、同一场景下的统计，才能支持“官方工具和 ESP 算法差异”的结论。

## 15. 当前问题清单与风险等级

### P0/P1：必须先澄清的比较风险

| 风险 | 等级 | 处理建议 |
|---|---|---|
| 把官方槽号当人物 ID | P1 | 官方和 ESP 日志同时显示 `slot` 与 `track_id`，禁止混用 |
| 把 raw count 与 confirmed count 比较 | P1 | 对比表固定同时记录四层数量 |
| C5 BLE UUID/MAC 未经真机确认 | P1 | 维持 fail-closed；完成 GATT discovery 后再启用 |
| 把工具显示稳定当成官方算法证明 | P1 | 只写可见行为，不猜 EXE/DLL 内部实现 |
| UART 离线被当成无人 | P1 | 保持 ESP `UNKNOWN/OFFLINE` 语义 |

### P2：现场精度和体验风险

| 风险 | 等级 | 处理建议 |
|---|---|---|
| 安装轴向/原点未标定 | P2 | 用已知点位标定 flip/rotation/origin |
| 官方模块过滤和 S3 zone 同时修改 | P2 | 先关闭一层做基线，再逐层启用 |
| 多人交叉导致 track ID 交换 | P2 | 使用真机交叉回放统计 ID switch |
| 静止目标和多径反射 | P2 | 记录 raw/accepted/velocity outlier，按场景调阈值 |
| 旧文档参数与当前代码不一致 | P2 | 以 `radar_config.h` 和当前源码为准，历史报告只作背景 |

## 16. 最终结论与下一步

### 最终结论

1. 官方 LD2450 工具是优秀的联调、配置和当前目标可视化工具，但从可读源码不能证明它提供了稳定的人体身份轨迹或成熟的人数业务状态机。
2. 官方协议只给最多三个目标槽和位置/速度/分辨率，主控侧必须自己解决“跨帧身份、人数去抖、失帧、区域和业务语义”。
3. 当前 ESP 项目在数据接收、错误恢复、多源传输、空间层、轨迹层和业务状态层明显比官方 STM32 参考工程完整；它的 `track_id`、confidence、occupancy 和 motion 都是 ESP 软件层新增语义，不是 LD2450 原生字段。
4. ESP 项目不能直接声称“比官方工具更准”。目前能证明的是软件架构和 host replay 覆盖更完整；真实精度和官方工具一致性必须通过同源原始帧 + 同场景硬件测试证明。

### 推荐下一步顺序

1. 用官方 Windows/Android 工具读取并记录当前 LD2450 固件版本、单/多目标模式和区域过滤配置。
2. 对真实雷达完成 BLE Service/Notify/Write discovery，确认地址类型和 MAC；在此之前保持 C5 BLE 禁用。
3. 让官方工具、S3 UART/C5 BLE 和 macOS Debugger 同时记录同一时间窗。
4. 先比较 raw frame 解码一致性，再比较 accepted、track、人数和运动状态，最后才调阈值。
5. 用上述场景矩阵形成“官方显示 vs ESP raw/accepted/confirmed”的硬件验收报告。
