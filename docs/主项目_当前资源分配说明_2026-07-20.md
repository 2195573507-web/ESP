# 主项目当前资源分配说明

日期：2026-07-20

## 1. 范围、口径与结论

本文只覆盖工作区顶层主项目中的 `ESPC51`、`ESPC52`、`ESPS3` 与 `ESP-server`。
`分支项目/`、`archive/`、构建目录中的历史配置和已删除文件均不作为当前资源事实来源。

“资源分配”分为四层：板级职责、通信与数据所有权、FreeRTOS 任务/队列、内存与静态镜像。
源码能确认配置值和分配策略；现有 S3 map 能确认一次已生成镜像的静态占用；动态堆余量、PSRAM
余量、CPU 占用、任务高水位和外设并发稳定性必须以真机日志为准，本文不将其写成已验证结论。

当前主项目采用 **双 C5 端采集/交互 + S3 网关集中接入和调度 + ESP-server 外部业务服务** 的形态。
S3 是 C5 的本地协议终点和 Server 的唯一网关；C5 不应承担 HOME 空间解释、Server 聚合或 LLM
职责。S3 的可选能力（本地雷达、habit、环境告警、语音代理）均按模块失败隔离；其中 C5 启动任务
创建失败仍会进入 `ESP_ERROR_CHECK`，与 S3 的降级策略不同。

## 2. 板级职责与通信边界

| 资源主体 | 已分配职责 | 输入 | 输出/所有权 | 不承担的职责 |
| --- | --- | --- | --- | --- |
| ESPC51 | C5 终端：Wi-Fi 接入 S3、BME/本地传感、麦克风/扬声器、BLE 雷达采集及轻量状态上传 | 本地 I2C/IIS/MIC、BLE、S3 SoftAP | 向 S3 的 `/local/v1` 设备协议、语音 turn 和雷达结果 | HOME 聚合、Server 完整协议、空间坐标解释 |
| ESPC52 | 与 C51 同一资源模型的第二 C5 终端，保留独立设备绑定和本地外设实例 | 同上 | 同上 | 同上 |
| ESPS3 | 网关：SoftAP/Wi-Fi、子设备登记、本地 HTTP、雷达 UART/空间域、队列调度、上行网络、命令、BME 缓存与语音代理 | C5 本地 HTTP/UDP、S3 本地 LD2450、Server 响应 | C5 协议在本地结束；`server_client` 负责 S3 到 ESP-server 的完整 HTTP | 将 C5 工作直接转嫁给 ESP-server；把雷达解析塞进 HTTP 回调 |

代码边界证据：S3 `main` 只创建启动任务，随后交给 `gateway_orchestrator`；本地 C5 协议由
`local_http_server/protocol_adapter` 结束，而 `server_client` 只面向 S3 到 Server
（`ESPS3/main/main.c:25-84`、`ESPS3/components/Middlewares/server_client/server_client.c:1-12`）。

### 2.1 实际数据路径

```text
C51/C52 本地传感、BLE 雷达、MIC/IIS
        |  轻量本地 Wi-Fi 协议
        v
ESPS3 local_http_server / protocol_adapter / radar_ingest
        |  S3 内部队列、聚合、调度、缓存
        v
ESPS3 server_client / network_worker
        |  完整 HTTP
        v
ESP-server / SQLite / Agent 与管理 API
```

S3 本地 LD2450 是另一条输入，直接进入 `radar_rx -> radar_local -> RadarSourceState /
RadarRoomState / RadarHomeState`。它与 C5 雷达上报在 S3 汇合，但 C5 槽位不能被解释为跨帧或跨源的
人体身份。

## 3. 启动资源的取得顺序

S3 的启动次序不是单纯的函数排列，而是资源依赖顺序：

1. 创建 `gateway_startup`，栈 `8192 B`、优先级 `4`，且显式要求 internal 8-bit RAM，保证
   Flash/NVS cache-off 路径安全。
2. 启动 Wi-Fi/event loop；随后初始化 resource manager 和 scheduler 的基础队列，防止网络回调在
   队列出现前发布状态。
3. 创建 network worker 并请求 local HTTP；再初始化雷达 registry/ingest/local adapter、habit
   规则/上报和 BME 缓存。
4. 初始化轻量状态服务、雷达诊断、环境告警和 scheduler；最后启动 reporter/replay/device stream，
   语音代理刻意最后启动。

各阶段前后记录 `STARTUP_MEMORY_CHECK`，记录 internal 可用量、最大连续块和 PSRAM 可用量；某个
模块返回错误时会输出 `module disabled` 并继续互不依赖的模块。来源：
`ESPS3/main/main.c:62-84`、
`ESPS3/components/Middlewares/gateway_orchestrator/gateway_orchestrator.c:40-151`。

C51/C52 则创建 `app_startup`（`12288 B`、优先级 `3`），启动链是 Wi-Fi -> register -> BME ->
voice -> command。当前 C5 `app_main` 的启动任务创建失败调用 `ESP_ERROR_CHECK(ESP_ERR_NO_MEM)`，
因此它不是 S3 那种“仅禁用一个可选模块”的同等级降级点
（`ESPC51/main/main.c:23-78`、`ESPC52/main/main.c:23-78`）。

### 4.1 S3 调度和背压边界

- scheduler 基础 tick 为 `100 ms`，软/硬水位分别是 `5/9`；两条 worker 队列均为 `12` 项，
  不是无限队列。
- network worker 工作/命令队列深度均为 `16`，断开事件队列以最大子设备数为上限。
- voice proxy 只允许一个待处理 session；habit event reporter 只保留一个 `s_pending` 事件并将
  重试间隔限制为 `1/2/5/10/30 s`。
- 环境告警的历史和 reporter FIFO 与主网络 worker 解耦，避免告警重试占住 network/voice 主路径。

这些是设计上的容量上限，不代表在离线、满队列或 Server 超时场景已完成真机压测。

## 6. 内存分配策略

### 6.1 Internal RAM：为不可替代路径保留

S3 和 C5 均设置 `CONFIG_SPIRAM_MALLOC_RESERVE_INTERNAL=32768`，且小于 `16384 B` 的默认 malloc
优先留在 internal RAM。S3 启动任务显式为 internal RAM；Wi-Fi/lwIP、DMA/IIS、FreeRTOS 的控制
对象和 cache-off 相关路径不能被笼统迁往 PSRAM。

这意味着“有 PSRAM”不等于 internal RAM 不再紧张：S3 的 HTTP、Wi-Fi、命令与音频同时活跃时，
最大连续 internal block 比总 free size 更关键。启动期已记录两项指标，但需要设备串口日志才可判定
是否够用。

### 6.2 PSRAM：大栈、队列、工作区和历史数据

当前明确放入 PSRAM 的资源包括：

- S3 network worker、scheduler worker、雷达 RX/local/diag/log、BME replay、device stream、
  voice proxy 的任务栈；network/scheduler 队列也使用 `xQueueCreateWithCaps`。
- S3 环境告警 history、reporter 存储和 completion queue。
- C5 雷达 history/上传 JSON、雷达 worker/upload/BLE 任务栈、MIC 任务栈和 IIS writer 静态栈。

这是一项“能力请求”而非实测使用量。每次 `heap_caps_*` 或 `xTaskCreateWithCaps` 失败都需要按模块
停止、释放已取得的资源并记录错误，不能用临时增加栈或降低 internal reserve 掩盖问题。

### 6.3 当前可量化的静态镜像基线

## 7. 资源风险与维护规则

1. 不把 HTTP handler、UART RX callback 或 BLE notify callback 当作解析、JSON 构造或上行重试 worker；
   它们只入队/投递，耗时工作交给独立任务。
2. 不修改 internal reserve、PSRAM 任务栈、Wi-Fi/IIS DMA 分配或任务优先级来“给新功能腾空间”，除非
   同时提供 map、启动 heap、任务 high-water、离线队列和并发外设的证据。
3. `source_person_count` 和 `home_person_count` 分别属于源与 HOME 汇总语义；HOME 必须从当前在线
   source context 重算，不能由 `active_source` 或缓存房间状态反推。
4. C51/C52 保持资源模型对等，但不能覆盖各自身份、绑定、房间与硬件引脚；它们不是可互换的同一个
   运行实例。
5. ESP-server 的 Node 进程、SQLite、LLM/天气 HTTP 调用不在 ESP RAM 预算内。其并发、认证、磁盘和
   外部 API 限额应作为服务端资源表独立管理，不能用固件堆统计代替。
