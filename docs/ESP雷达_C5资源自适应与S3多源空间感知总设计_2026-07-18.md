- 文档日期：2026-07-18

- 适用目标：ESPC51、ESPC52、ESPS3、ESPS3-Radar-Debug

- 当前阶段：C52 BLE 雷达真机接收与解析已验证；后续进入资源调度、三源独立日志和空间感知收敛阶段

---

## 1. 文档目标

本设计在不推翻现有雷达链路的前提下，实现以下目标：

1. C5 根据人体状态和语音状态自适应分配 CPU、网络、队列和上传资源。
2. 无人时降低雷达上报开销，提高环境监测价值。
3. 有人尤其有人运动时，优先保证雷达数据的新鲜度、目标字段完整性和连续性。
4. 语音始终为 C5 最高优先级；唤醒后继续保持既有独占语义。
5. S3 对 `S3_LOCAL`、`C51`、`C52` 三个雷达源独立处理、独立限频、独立输出。
6. S3 空间感知按“每个房间独立建模，家庭级汇总”实现，不做无依据的跨房间目标融合。
7. 冻结现有 S3 雷达日志契约；若确需改变日志结构，必须在同一变更中同步修改 Mac 本地解析工具和测试样本。
8. 不影响既有 voice、BME690、command、Wi-Fi、S3 本地 UART 雷达和 C5 BLE 雷达基础链路。

## 2. 当前已验证基线

### 2.1 C5 BLE 雷达链路

当前 C51/C52 源码采用固定 MAC 白名单连接各自的 LD2450：

| 终端 | local_id | device_id | 雷达 MAC |
|---|---:|---|---|
| C51 | 1 | `sensair_shuttle_01` | `C1:BC:3C:3C:3D:60` |
| C52 | 2 | `sensair_shuttle_02` | `8C:B1:F3:E1:15:41` |

C52 真机日志已经确认：

- 发现 `FFF0` 服务；
- 发现 `FFF1` Notify 特征；
- 发现 `FFF2` Write Without Response 特征；
- 对 `FFF1` 的 CCCD 写入成功；
- 收到真实 Notify；
- 30-byte LD2450 帧解析成功；
- filter 后有效输出约 11–12 Hz；
- Notify 约 22–24 次/秒，约两个 Notify 片段组成一帧有效 30-byte 数据；
- parser/filter 队列无明显积压，窗口内 `drop_count=0`。

因此，后续不得再把 C5 BLE 问题描述为“未接入”或“缺少启动命令”。官方 iOS 示例和真机结果均表明，LD2450 正常路径是订阅 `FFF1` 后直接接收数据；`FFF2` 保留为配置通道，不应发送猜测命令。

### 2.2 C5 当前处理与上传

当前链路：

```text
LD2450 BLE Notify
  -> C5 BLE byte stream
  -> 30-byte parser
  -> C5 edge filter
  -> p=3 radar JSON
  -> POST /local/v1/radar/result
  -> S3
```

当前上传机制为“有效帧驱动”：每产生一份有效 filter 输出，就尝试生成并上传一份雷达结果。若有效帧约 11–12 Hz，则理论上传尝试也可达到约 11–12 次/秒。

当前协议必须保持：

```json
{
  "p": 3,
  "id": 2,
  "t": "radar",
  "u": 123456,
  "q": 10,
  "v": {
    "device_id": "sensair_shuttle_02",
    "link_state": 5,
    "sample_valid": 1,
    "frame_seq": 42,
    "frame_uptime_ms": 123450,
    "target_count": 1,
    "targets": [
      {
        "target_id": 1,
        "x_mm": 100,
        "y_mm": 2000,
        "velocity_cm_s": -15,
        "confidence": 70,
        "resolution_mm": 200,
        "distance_mm": 2002
      }
    ]
  }
}
```

注意：C5 的 `target_id` 是 LD2450 slot 标识，不是跨帧稳定人体身份。稳定 `track_id` 由 S3 tracker 生成。

### 2.3 S3 当前多源链路

S3 冻结的 source ID：

| Source | ID | 入口 |
|---|---:|---|
| `S3_LOCAL` | 0 | S3 UART1 LD2450 |
| `C51` | 1 | C51 `POST /local/v1/radar/result` |
| `C52` | 2 | C52 `POST /local/v1/radar/result` |

当前 S3 设计已经具备：

- 每个远端 source 独立 latest-only pending sample；
- 每个 source 独立空间状态实例和 tracker；
- 坐标变换、有效范围过滤、EMA、跳变抑制、最近邻关联；
- 轨迹确认、hold、删除、occupancy、motion、stale/offline；
- `GET /local/v1/radar/debug` 本地调试接口；
- S3 本地 UART 独立恢复状态机。

本设计不重写这些算法模块，只做配置统一、source 隔离、日志调度和必要的空间语义收敛。

## 3. 总体设计原则

### 3.1 优先级顺序

C5 运行时优先级必须固定为：

```text
VOICE_ACTIVE
    >
RADAR_ACTIVE
    >
SYSTEM / COMMAND / LINK HEALTH
    >
BME REPORTING
    >
BACKGROUND DIAGNOSTICS
```

这里的“优先级”不仅是 FreeRTOS task priority，还包括：

- 是否允许发起 HTTP；
- 队列容量和丢弃策略；
- 是否执行昂贵编码；
- 任务唤醒频率；
- 日志输出频率；
- 重试是否允许占用网络窗口。

### 3.2 采集、处理、上传、日志必须分开

禁止继续把以下四个频率视为同一个概念：

1. **采集频率**：BLE Notify 到达频率。
2. **处理频率**：parser/filter 实际处理频率。
3. **上传频率**：C5 向 S3 发送最新雷达状态的频率。
4. **日志频率**：C5/S3 为人类调试输出的频率。

原则：

- 采集和处理尽量连续，保证状态机有足够数据；
- 上传根据状态自适应；
- 日志独立限频，绝不能反向影响算法；
- 无人状态降低的是上传与日志，不是随意关闭 BLE 或破坏 parser 连续性。

### 3.3 雷达“完整性”的定义

本系统优先保证的是：

- 每次上传的目标字段完整；
- 最新目标状态不会被旧帧阻塞；
- 状态变化和运动阶段具有足够连续样本；
- S3 能通过 `frame_seq`/`q` 识别跳帧。

本系统不承诺通过 HTTP 无损保存每一帧历史数据。若要求逐帧零丢失，需要持久化队列、流协议和更高网络预算，这与“语音最高优先级”和“最新位置优先”冲突。

因此，雷达上传采用**最新状态优先、状态变化立即、移动阶段高频、旧帧可合并**的策略。

## 4. C5 自适应资源状态机

### 4.1 状态定义

C51/C52 使用同一套状态机，除身份配置外源码必须保持一致。

```c
typedef enum {
    C5_SENSE_VACANT_STABLE = 0,
    C5_SENSE_VACANT_RECENT,
    C5_SENSE_PRESENT_STILL,
    C5_SENSE_PRESENT_MOVING,
    C5_SENSE_RADAR_DEGRADED,
    C5_SENSE_VOICE_ACTIVE,
} c5_sense_mode_t;
```

#### `VACANT_STABLE`

- 连续无有效目标达到稳定无人门限；
- 雷达 BLE 连接保持；
- parser/filter 保持运行；
- 雷达上传降频；
- BME 报告频率提高。

#### `VACANT_RECENT`

- 刚从有人转为空，或刚启动但已有有效空帧；
- 需要较高频率确认“离开”不是短时漏检；
- 维持短时间后进入 `VACANT_STABLE`。

#### `PRESENT_STILL`

- 已确认目标存在；
- 运动证据不足或已退出 moving；
- 雷达上传中频，维持存在和位置；
- BME 降为中低频。

#### `PRESENT_MOVING`

- 已确认目标运动；
- 雷达上传升到高频；
- 雷达 HTTP 进入高优先级通道；
- BME 上传显著降频；
- 诊断日志仍限频，不能逐帧淹没串口。

#### `RADAR_DEGRADED`

- BLE 断开、Notify 长时间无数据、parser 连续错误或数据 stale；
- 立即上传一次健康变化；
- 进入有界重连；
- 不以“无目标”代替“传感器不可用”。

#### `VOICE_ACTIVE`

- WakeNet 唤醒后立即进入；
- 语音拥有 CPU、网络、DMA、内部 RAM 和任务调度最高优先级；
- BME 主动上传暂停或极低频；
- 雷达保持 BLE 连接和轻量接收，不发起高频 HTTP；
- 仅保留最新雷达状态，并以低频或关键状态变化方式上报；
- 语音结束后先上传一份最新雷达快照，再按雷达状态恢复对应频率。

### 4.2 状态优先级和覆盖关系

状态选择按以下顺序执行：

```text
if voice_active:
    VOICE_ACTIVE
else if radar_link_unhealthy or radar_sample_stale:
    RADAR_DEGRADED
else if moving_confirmed:
    PRESENT_MOVING
else if target_present:
    PRESENT_STILL
else if recently_vacated:
    VACANT_RECENT
else:
    VACANT_STABLE
```

不得让 BME 状态、网络扫描状态或日志状态覆盖语音/雷达状态。

### 4.3 推荐初始门限

以下是首轮工程默认值，必须集中配置，不得散落硬编码：

| 参数 | 推荐默认值 | 说明 |
|---|---:|---|
| 有人确认 | 连续 2 个有效输出 | 与 S3 两帧确认思路一致 |
| moving 进入 | 连续 2 帧满足运动证据 | 避免单帧噪声 |
| moving 退出 | 1500 ms 无运动证据 | 加滞回避免抖动 |
| recent vacant 持续 | 10 s | 防止刚离开时过早降频 |
| stable vacant 进入 | 3 s 无目标且 sample fresh | “无目标”必须基于有效帧 |
| radar stale | 1000 ms 无有效 parser 输出 | 与 C5 当前 sample 失效边界对齐 |
| voice 恢复冷却 | 2000 ms | 避免语音结束瞬间网络突发 |
| 运动速度门限 | 16 cm/s 起步值 | 后续按实机数据校准 |
| 坐标运动门限 | 500 ms 内位移 200 mm | 与速度证据并行使用 |

运动判定不应只依赖 LD2450 的单一速度字段，建议：

```text
moving_evidence =
    abs(filtered_velocity_cm_s) >= velocity_threshold
    OR
    displacement_mm_within_window >= displacement_threshold
```

### 4.4 C5 推荐上传频率

| C5 状态 | 雷达处理 | 雷达上传到 S3 | BME 上传 | 日志 |
|---|---|---:|---:|---:|
| `VACANT_STABLE` | 保持原始有效帧处理 | 0.2 Hz（5 s 一次） | 0.1 Hz（10 s 一次）或现有较高环境频率 | 0.1–0.2 Hz |
| `VACANT_RECENT` | 全速 | 1 Hz | 0.05 Hz（20 s） | 1 Hz |
| `PRESENT_STILL` | 全速 | 2 Hz | 0.02–0.033 Hz（30–60 s） | 1–2 Hz |
| `PRESENT_MOVING` | 全速 | 10 Hz 上限，跟随有效 filter 输出 | 0.016 Hz（60 s） | 1 Hz 摘要，必要时 5 Hz 临时调试 |
| `RADAR_DEGRADED` | 恢复/重连 | 状态变化立即 + 0.2 Hz 健康心跳 | 保持低频 | 变化立即 + 0.2 Hz |
| `VOICE_ACTIVE` | 保持 BLE/轻量 parser，latest-only | 1 Hz 上限；关键 presence 变化可立即合并上报 | 暂停主动上传，结束后补一份最新值 | 0.2–1 Hz |

说明：

- `PRESENT_MOVING` 的 10 Hz 是上传上限，不要求制造不存在的帧；实际上传频率不得超过有效 filter 输出。
- BME 的“上传频率”不等于传感器采样频率。BME690 的内部采样、基线维护和 IAQ 算法所需周期应保持稳定，除非有单独的传感器算法审计证明可以改变。
- 无人时提高的是 BME 的**有效报告和环境分析频率**，不应盲目重启传感器或破坏 baseline。

## 5. C5 上传调度器设计

### 5.1 不再采用逐帧排队重试

当前“每个有效帧入上传队列 + HTTP 重试”在 10–12 Hz 输入下容易产生：

- 网络故障时积压旧坐标；
- 3 秒 HTTP 超时放大阻塞；
- 多次重试与新帧竞争；
- 语音期间网络和内部内存压力；
- 上传成功后数据已经过时。

改为：

```text
filter output
  -> latest_radar_snapshot（始终覆盖最新）
  -> adaptive upload scheduler
  -> one in-flight HTTP request
  -> one latest pending snapshot
```

只允许：

- 1 个正在发送的雷达请求；
- 1 个待发送最新快照；
- 新帧到来覆盖 pending 旧帧；
- 绝不排队保存大量旧位置。

### 5.2 优先队列

网络发送分为逻辑优先级：

```text
P0: voice transport
P1: radar state transition / moving snapshot
P2: system register / heartbeat / command response
P3: BME report
P4: diagnostics / background
```

这里不要求建立五个物理队列，可在现有 scheduler 中增加 priority 和 coalescing key：

- `voice:*` 不合并；
- `radar:<local_id>` latest-only；
- `bme:<device_id>` latest-only；
- heartbeat 可合并；
- command response 不得被普通 telemetry 覆盖。

### 5.3 雷达上传触发

满足任一条件立即调度：

1. `target_count` 从 0 变为 >0；
2. `target_count` 从 >0 变为 0；
3. 状态从 still 变 moving；
4. 状态从 moving 变 still；
5. link/sample health 变化；
6. 当前模式的周期到期；
7. 语音结束，需要补发最新快照。

普通位置更新受状态频率上限控制。

### 5.4 失败和重试

雷达数据不能沿用“旧请求反复重试直到成功”的语义。

推荐：

- HTTP 失败后记录失败计数；
- 若已有更新快照，下一次重试直接发送最新快照；
- 重试退避建议 100/250/500/1000 ms，上限 1 s；
- voice active 时不执行密集重试；
- 连续失败后进入 link degraded，只保留 1 份 latest；
- 恢复成功后立即发送 latest，不补发历史帧。

必须保留以下计数：

```text
radar_generated_count
radar_upload_attempt_count
radar_upload_success_count
radar_upload_failure_count
radar_upload_coalesce_count
radar_upload_stale_skip_count
radar_state_transition_upload_count
```

### 5.5 C51/C52 一致性

C51/C52 只允许以下差异：

- local ID；
- device ID；
- radar ID；
- MAC；
- room binding（且必须与 S3 registry 一致）。

调度器、状态机、频率、日志、测试代码必须字节级一致或通过模板/共享组件生成。

## 6. C5 与语音/BME 协作

### 6.1 语音独占

WakeNet 触发后：

1. 标记 `voice_active=true`；
2. 禁止 BME 主动 HTTP 上传；
3. 雷达 HTTP 降至 1 Hz latest-only；
4. 雷达 BLE callback 只做有界复制；
5. parser/filter 可继续轻量运行，不能 malloc 大对象；
6. 禁止雷达日志逐帧输出；
7. voice 完成后进入 2 s 冷却；
8. 冷却结束发送最新 radar + BME 快照各一份；
9. 恢复当前人体状态对应频率。

若内部 RAM、DMA 或网络不足，牺牲顺序为：

```text
background logs
 -> BME upload
 -> radar ordinary periodic upload
 -> radar processing detail
 -> voice（不得牺牲）
```

### 6.2 BME 策略

BME 分成三层：

1. **传感器测量**：按 BME690 算法和 baseline 要求运行；
2. **本地状态更新**：可保持正常；
3. **网络上传**：由资源状态机自适应。

不得为了雷达简单停掉 BME driver 或反复重新初始化，这会破坏基线和长期稳定性。

## 7. S3 三源独立处理

### 7.1 source 独立性

以下状态必须按 source 分开保存：

```c
typedef struct {
    radar_source_id_t source;
    radar_spatial_state_t spatial;
    radar_tracker_t tracker;
    radar_source_health_t health;
    radar_log_schedule_t log_schedule;
    radar_latest_snapshot_t latest;
} radar_source_runtime_t;
```

每个 source 独立维护：

- last frame time；
- last valid time；
- tracker IDs；
- occupancy；
- motion；
- hold/stale/offline；
- last emitted state；
- last track log time；
- last diagnostic log time；
- output frequency window；
- drop/sequence counters。

禁止共享单一：

```text
last_state
last_log_time
last_track_id
last_source
```

否则 `S3_LOCAL` 的高频状态会压制 C51/C52 输出。

### 7.2 算法频率和日志频率分离

S3 ingest 和空间算法处理所有可接受输入，不因日志降频而降频。

```text
input frame -> tracker/spatial every accepted frame
                           |
                           +-> adaptive log scheduler
```

日志调度只控制串口输出，不控制：

- 坐标变换；
- tracker update；
- occupancy/motion 状态机；
- registry 更新；
- `/local/v1/radar/debug` 数据。

### 7.3 S3 每源推荐日志频率

| Source 状态 | `RADAR_TRACK` 类 | 状态摘要 | 健康摘要 |
|---|---:|---:|---:|
| moving | 5–10 Hz | 状态变化立即，稳态 2 Hz | 1 Hz |
| present/still | 2 Hz | 状态变化立即，稳态 1 Hz | 0.5 Hz |
| recent vacant/hold | 1 Hz | 状态变化立即 | 0.5 Hz |
| vacant stable | 0.2–0.5 Hz | 状态变化立即 | 0.2 Hz |
| stale/offline/recovery | 无坐标则不打印 track | 状态变化立即 | 0.5 Hz，恢复期间有界 |

三个 source 的计时器互不影响。例如 C52 moving 10 Hz，不得提高或降低 C51 和 S3_LOCAL 的输出。

## 8. S3 日志兼容性硬约束

### 8.1 日志即接口

当前 Mac 本地雷达工具依赖 S3 串口日志解析，因此 S3 雷达日志属于兼容性接口，不是可随意修改的调试文本。

必须冻结现有主要标识和字段语义，包括但不限于：

```text
RADAR_STATE
RADAR_TRACK
RADAR_TRACK_COMPAT
RADAR_TRACK_COMPAT_METERS
RADAR_TRACKER
source transition source=
local transition sensor=
local track=
```

### 8.2 允许的变更

允许：

- 在行尾追加新字段；
- 增加新的独立摘要行；
- 增加 `source=`、`room=`、`hz=`、`age_ms=` 等可选字段；
- 对不同 source 调整输出时间，不改变行结构。

不允许：

- 重命名已有日志标识；
- 删除已有必要字段；
- 改变字段单位；
- 将 `x/y` 从 mm 改为 m 而不改字段名；
- 改变 `local track=` 的基本语法；
- 只修改 S3 而不验证 Mac parser。

### 8.3 必须同步修改 Mac 工具的情形

- 改变日志 tag；


### 8.4 日志兼容测试门禁

1. S3_LOCAL moving；

### 8.5 长期方向

## 9. 空间感知最优方案

### 9.1 不做跨房间雷达融合

当前三个雷达对应不同房间。最佳方案不是把三台雷达的坐标放入一个全局 tracker，而是：

```text
S3_LOCAL -> Room A Spatial Context
C51      -> Room B Spatial Context
C52      -> Room C Spatial Context
```

每个房间独立进行：

1. 坐标归一化；
2. 房间边界裁剪；
3. target association；
4. stable track；
5. zone；
6. occupancy；
7. motion；
8. freshness/health。

家庭级只汇总房间状态，不直接融合目标坐标。

### 9.2 每房间状态模型

传感器健康、占用和运动必须分离：

```c
typedef struct {
    radar_sensor_health_t sensor;   // VALID / STALE / OFFLINE / RECOVERING
    radar_occupancy_t occupancy;    // UNKNOWN / VACANT / HOLD / PRESENT
    radar_motion_t motion;          // UNKNOWN / NONE / STILL / MOVING
} radar_room_state_t;
```

禁止用 `target_count=0` 直接推导：

- 传感器在线；
- 房间必定无人；
- tracker 应立即删除；
- 最近目标坐标应清空。

正确语义：

- sample fresh 且无目标：可逐步进入 vacant；
- sample stale/offline：occupancy 为 unknown，不是 vacant；
- 短暂丢目标：进入 hold；
- 目标恢复：继续或重建 track，避免瞬时跳变。

### 9.3 空间处理流水线

推荐固定顺序：

```text
source sample
 -> schema/identity/freshness validation
 -> coordinate transform
 -> room bounds filter
 -> target validity and outlier gate
 -> nearest-neighbor association
 -> track confirmation
 -> EMA / velocity / jump suppression
 -> zone assignment
 -> occupancy and motion state machine
 -> registry/debug/log output
```

### 9.4 Zone 必须真正接入

当前 zone map 已初始化但部分 frame path 未传入 tracker。后续若要形成“空间感”，必须把正确的 per-source zone map 传入 tracker/空间状态，而不是只保留配置文件。

首期不建议复杂划区，先实现：

- 每个房间一个 full-room active zone；
- 一个 near-radar exclusion zone（可选）；
- 按实测位置再增加 bed/desk/door 等语义区。

Zone 不能依赖未经验证的坐标方向。必须先完成已知物理位置校准：左、中、右、近、远。

### 9.5 家庭级汇总

新增轻量 home presence aggregator：

```json
{
  "rooms": {
    "living_room": {"occupancy":"present", "motion":"moving"},
    "bedroom": {"occupancy":"vacant", "motion":"none"},
    "s3_local": {"occupancy":"unknown", "motion":"unknown"}
  },
  "occupied_room_count": 1,
  "active_room": "living_room",
  "last_transition": "bedroom->living_room"
}
```

第一阶段不做跨房间 person ID。可在未来基于房间邻接、时间窗口和门口 zone 推断房间转移，但不得宣称是同一个人身份识别。

### 9.6 参数集中化

以下参数必须从一个权威配置来源读取：

- association gate；
- confirm frames；
- hold time；
- delete time；
- stale/offline；
- EMA alpha；
- jump/velocity gate；
- motion enter/exit；
- room bounds；
- zone map；
- per-source axis flip/rotation/offset。

禁止 `radar_config.h` 与 `radar_spatial_state.c` 使用不一致的硬编码默认值。

## 12. 故障与降级策略

### 12.1 C5 BLE 断开

- 进入 `RADAR_DEGRADED`；
- 立即上传健康变化（若 S3 link 可用）；
- 指数退避重连；
- 不把离线误报为 vacant；
- BME 可恢复到无人环境报告策略；
- voice 不受影响。

### 12.2 C5 到 S3 HTTP 异常

- latest-only 合并；
- 不积压 32 个旧位置；
- 不重复重试过时帧；
- link 恢复后发送最新状态；
- 记录 sequence gap 和 coalesce。

### 12.3 S3 source stale/offline

- 仅更新该 source；
- 不清空其他房间；
- Mac 工具显示该面板离线，保留最后坐标并标注 stale，直到过期策略清除；
- S3_LOCAL UART recovery 不影响 C51/C52；
- C51 断开不影响 C52。

### 12.4 语音期间

- voice 不等待 radar/BME；
- radar/BME 不得持有 voice 需要的锁、网络 worker 或大块内部 RAM；
- voice 结束后有序恢复，禁止多个 telemetry 同时突发。

## 16. 验收标准

### 16.1 C5 功能

- C51/C52 BLE 在各自 MAC 下稳定连接；
- 实际 parser/filter 频率保持约设备输出水平；
- `PRESENT_MOVING` 时雷达上传达到 8–10 Hz 或实际有效帧上限；
- `PRESENT_STILL` 时约 2 Hz；
- `VACANT_STABLE` 时约 0.2 Hz；
- 状态变化上传延迟 < 200 ms（网络正常）；
- 语音 active 时 radar 普通上传不超过 1 Hz，BME 不主动抢占；
- voice 结束后 2 s 内补发最新 radar/BME；
- 连续 10 分钟 moving 不产生上传队列增长；
- HTTP 失败时 pending 始终是最新样本；
- C51/C52 parity PASS。

### 16.2 S3 功能

- `S3_LOCAL`、`C51`、`C52` 独立状态；
- 一个 source 高频 moving 不压制其他 source 日志；
- 状态变化立即输出；
- 稳态日志达到配置频率，误差不超过一个周期；
- tracker/空间算法处理频率不受日志限频影响；
- source stale/offline 只影响本 source；
- zone path 实际生效；
- 配置值与算法使用值一致。

### 16.3 Mac 工具

- 旧日志样本继续解析；
- 三源交错日志正确分板；
- 每个 source 坐标、状态、在线状态独立；
- S3_LOCAL 日志不会覆盖 C51/C52；
- C51/C52 无坐标更新时不会因健康日志清空有效坐标；
- 若日志结构未改，Mac 工具无需修改；
- 若日志结构改动，S3 + Mac 同一提交并全部测试通过。

### 16.4 资源

- voice 识别、录音、播放无新增退化；
- internal heap 和 largest block 不低于现有安全线；
- radar/BME/voice task watermark 无接近栈溢出；
- radar moving 状态下 event queue 不持续增长；
- telemetry 失败不造成 watchdog、死锁或重启。

## 19. 最终结论

本系统的最佳方向不是简单地把所有频率降低，也不是让 C5 或 S3 单独承担全部智能，而是：

```text
C5：连续采集 + 轻量处理 + 资源感知的最新状态上报
S3：三源独立 tracker + 每房间空间状态 + 家庭级汇总
Mac：兼容日志契约下的三源独立可视化
Voice：任何时候最高优先级
```

该方案保留现有已验证链路，同时解决：

- 无人时不必要的雷达 HTTP 压力；
- 有人时雷达实时性不足；
- BME 与雷达资源竞争；
- voice 唤醒后的资源独占；
- S3_LOCAL/C51/C52 日志互相压制；
- 多房间空间语义不清；
- S3 日志变化导致 Mac 工具失效。
